Lesson 5.1 — Security & Privacy Risks

Prompt Engineering and Privacy: What You Should Never Type Into an AI Tool at Work

Knowing how to write a good prompt is a skill. Knowing what not to put in a prompt is how you protect your career. This article covers the specific inputs that create legal and security exposure — with real before-and-after examples you can learn from.

The Prompt Is the Risk

When people think about AI security, they often picture someone hacking into a system or stealing files. But in most workplace AI incidents, the breach happens the other way: the employee hands the sensitive information to the AI tool willingly — inside the prompt.

A prompt is simply the text you type into an AI tool to get it to do something. It might be a question, a task, a document you paste in, or a combination of all three. Every word in that prompt travels to an external server. And every piece of sensitive information you include in the prompt goes with it.

This is why learning to write good prompts at work is really about two things: (1) getting useful output from the AI, and (2) doing it without creating a security or legal problem. This article focuses on the second part.

You don't have to choose between being productive with AI and being safe. You just have to learn how to give the AI what it needs to help you — without giving it what it doesn't need to know.

The Six Types of Prompts That Create Problems

These are the most common ways tech workers accidentally create risk when using AI tools. For each one, you'll see a risky version and a safer version that gets the same result:

❌ Risky Prompt
✅ Safe Prompt
"My script can't connect to the database. Here's my connection string: postgresql://admin:P@ssw0rd!2024@db.internal.mycompany.com:5432/production"
"My script can't connect to a PostgreSQL database. Here's a sample connection string with placeholder values: postgresql://username:password@hostname:5432/dbname. I'm getting a timeout error. What should I check?"
Why This Matters
The risky version includes a real username, a real password, and a real internal server address. All three are now on an external server. The safe version replaces these with generic placeholders — the AI can still help debug the connection issue using the fake values. The actual credentials never leave your environment.
❌ Risky Prompt
✅ Safe Prompt
"I need to write an email to our user Sarah Goldstein (sarah.g@clientcompany.com) about her account issue. Her account number is ACC-00492817 and she called in last Tuesday..."
"I need to write an email to a user who has an account issue. They called last week and weren't able to log in. The email should be professional and empathetic. I'll fill in the specific details after."
Why This Matters
The risky version sends real customer PII — name, email, and account number — to an external AI server. The safe version asks for the same email template without including any real person's details. You add the actual name, email, and account number yourself after the AI generates the template. The result is identical; the risk is gone.
❌ Risky Prompt
✅ Safe Prompt
"Here's our payment processing function. Why is it failing?

def process_stripe_charge(customer_id, amount):
  stripe.api_key = 'sk_live_abc123xyzREALKEY'
  charge = stripe.Charge.create(customer=customer_id...)"
"I have a Python function that calls a payment API. It takes a customer ID and amount, creates a charge, and returns a result. I'm getting a timeout on the API call. What are the most common causes of timeouts when calling third-party payment APIs, and how should I add retry logic?"
Why This Matters
The risky version includes a real API key and proprietary business logic. The safe version describes the problem in plain language — enough for the AI to give great debugging advice without ever seeing the actual code or credentials. For code review help, use generic or stripped versions, not production code.
❌ Risky Prompt
✅ Safe Prompt
"[Pastes full content of internal engineering roadmap document including Q4 product plans, unreleased features, and competitive strategy] Can you summarize this into bullet points?"
"I have an internal planning document that I need to summarize. The document covers quarterly product priorities, technical dependencies, and timeline. Can you give me a template or structure for a good technical roadmap summary, and I'll fill it in myself?"
Why This Matters
Internal strategy documents are confidential. If your company's unreleased product roadmap ends up in an AI tool's training data, it could — in theory — influence future outputs or be logged in systems you have no control over. Get the template or structure from the AI and do the summarizing yourself.
❌ Risky Prompt
✅ Safe Prompt
"Our server at 10.0.0.47 (staging-api.internal.mycompany.com) keeps throwing 502 errors when it connects to db-cluster-prod-01.us-east-1.internal. What's wrong?"
"An internal API server is throwing 502 errors when connecting to a database cluster in the same region. The connection works fine under low traffic but fails during peak load. What are the most common causes and how would you diagnose them?"
Why This Matters
Internal server hostnames and IP addresses reveal how your company's network is structured. A real internal hostname gives a potential attacker a roadmap. The safe version describes the problem functionally — the AI gives just as good a diagnosis without knowing your actual server names.
❌ Risky Prompt
✅ Safe Prompt
"Acme Corp (our biggest client) is upset because our API had downtime during their peak sales window last Friday. Their CTO, Marcus Webb, sent a strongly worded email. Help me write a response."
"Help me write a professional response to a client who experienced downtime during an important time window due to an issue on our end. The email should acknowledge the impact, take responsibility, explain what we've done to fix it, and offer a path forward. Tone should be professional and sincere."
Why This Matters
Client names, individual names, and details of client complaints are covered by your company's NDA and data agreements. The safe version produces the exact same email draft without exposing who the client is or the specifics of the situation. You add the client-specific details manually after.

The "Sanitize Before You Share" Checklist

Before you paste anything into an AI tool at work, run through this checklist. If you need to check more than one box, take a step back and remove those elements before proceeding:

🔍 Pre-Prompt Sanitization Checklist
✓
Remove all real names. Replace with "User A," "Client X," "Employee 1," or another placeholder. The AI doesn't need the real name to help you.
✓
Replace credentials with fake ones. If your code or config has a real password, API key, or token — swap it for "PLACEHOLDER_KEY" or "example_password_123" before sharing.
✓
Use generic table and variable names. Change "user_transactions" to "data_table." Change "patient_record_id" to "record_id." The AI can debug the logic without the real schema names.
✓
Remove internal URLs and server names. Replace "db.internal.company.com" with "database-server" or "hostname." Describe the environment type (production, staging, cloud) without naming your actual systems.
✓
Ask for structure, not the content itself. Instead of pasting a confidential document and asking for a summary, ask the AI for a summary template and fill it in yourself. You stay in control of what gets shared.
✓
Describe the problem, don't paste the evidence. Instead of sharing the full error log, describe what's happening. The AI can help you diagnose most issues from a description — and you avoid sharing internal system details.

The Quick Reference Rules

Memorize these six rules. They work across every type of AI tool and every work situation:

🚫
Never paste real credentials
Passwords and API keys. Ever. If you do it by accident, rotate the credential immediately — don't wait to see if anything happens.
👤
Replace real names with placeholders
Customer names, client names, employee names — swap them before pasting. The output quality is identical without the PII exposure.
🏗️
Use dummy schema names
Internal table names, column names, and database identifiers reveal system architecture. Replace with generic equivalents before asking for code help.
📝
Describe, don't paste
For internal documents and emails, describe what you need rather than pasting the original. The AI can generate a template you fill in yourself.
🌐
No internal server names
Internal hostnames, IPs, and staging URLs are sensitive. Describe the environment type without naming the actual systems.
✅
Use approved tools for sensitive work
Even with good sanitization habits, always use your company's approved AI tools when the task involves anything related to proprietary or regulated data.

📌 Article Summary

What to Remember

The prompt is the risk. Every word you type into an AI tool — including anything you paste — travels to an external server. Sensitive information in the prompt means sensitive information outside your control.

You can almost always get the AI's help without sharing the sensitive details. Describe the problem, use placeholder values, replace real names with generic ones, and ask for templates you fill in yourself. The AI's output is just as good — without the exposure.

The six things to never put in a prompt: real credentials, PII with real names, production source code with real schema names, internal documents pasted in full, internal server names and IPs, and client-specific details covered by your NDA.

If you accidentally paste a real credential into an AI tool, treat it as compromised immediately. Rotate the key or change the password before you do anything else. Don't wait to see if anything happens — the exposure already occurred the moment you pressed send.
