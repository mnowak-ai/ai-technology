Lesson 3.4 — Knowledge Sharing & Collaboration

Version Control for Documentation: Treating Docs Like Code

The same practices that keep code reliable — version control, peer review, automated checks, ownership — can keep documentation reliable too. Here's how the "docs as code" movement works and why it produces better documentation than wiki-first approaches.

Why Documentation Needs the Same Rigor as Code

Code has version control. Every change is tracked, attributed, and reversible. Code has peer review — before a change ships, another engineer reads it. Code has automated checks — CI pipelines catch broken builds before they reach production. Code ownership is explicit — someone is accountable for every service.

Most documentation has none of these things. It's edited anonymously, without review, with no automated quality checks, and with ownership that's either unclear or unenforceable. The result is documentation that's less reliable than the code it describes — which means engineers trust the code but can't trust the docs about it.

If code changes required no peer review, had no version history, and were managed by whoever wanted to edit them last, we'd call that a quality crisis. That's exactly how most organizations manage their documentation.

📄 The wiki approach (most teams)
How documentation typically works
• Anyone can edit any page at any time
• Changes have no review requirement
• No automated checks for accuracy or completeness
• Version history exists but nobody looks at it
• Ownership is assigned but not enforced
• Outdated content stays live until someone notices
💻 Docs as code approach
How documentation works when treated like code
• Changes go through pull requests with review
• CI checks flag broken links, deprecated references, missing fields
• Changes are attributed and fully reversible
• Ownership is tied to CODEOWNERS file — same as code
• Documentation lives in the same repo as the code it describes
• Stale docs trigger automated review requests

How Docs as Code Works in Practice

1
Documentation lives in the repository
Instead of a separate wiki, documentation files (usually Markdown) live in the same Git repository as the code they describe. A runbook for the inference API lives at /docs/runbooks/inference-api.md in the inference API repo. When the code changes, the docs change in the same PR.
my-service/ ├── src/ ├── tests/ └── docs/ ├── runbooks/ │ └── incident-response.md ├── architecture.md └── api-reference.md
2
Changes go through pull requests
A documentation change follows the same PR workflow as a code change. This enforces review: the engineer who owns the runbook reviews the PR before it merges. Changes are tracked, attributed, and reversible. The diff shows exactly what changed and why — the PR description provides context that a wiki edit history never captures.
# In the PR description: ## Documentation update Updated incident-response.md to reflect the new memory threshold for the rollback trigger (was 1.5GB, now 1.2GB based on Nov 15 incident findings). Also added the queue drain step that was missing. Fixes: P1-1115 action item #3
3
CI pipelines run automated checks on docs
Documentation CI checks catch errors that human review misses. These run on every PR that touches a documentation file — the same way tests run on every code PR.
4
CODEOWNERS ties documentation ownership to service ownership
The CODEOWNERS file — already used to require code review from specific engineers — extends to documentation. The person who owns the service must review changes to that service's documentation. Ownership is automatic and enforced, not voluntary.
# .github/CODEOWNERS docs/runbooks/inference-api.md @ml-team/inference-owners docs/runbooks/payment-service.md @payments-team/service-leads docs/architecture/*.md @platform-team/architects

What CI Checks Can Catch in Documentation

🔗
Broken internal links
A runbook that links to a deprecated Confluence page or a non-existent service name gets caught before the PR merges — not discovered by an engineer during a production incident.
📅
Stale review dates
A CI check that flags any runbook with a "last reviewed" date older than 90 days creates automated pressure to review documentation on a schedule — without requiring human tracking.
⚠️
References to deprecated services or commands
If a service or CLI command is sunset, a CI check can scan documentation for references to it and open issues automatically. Documentation that references deleted infrastructure is flagged before it causes an incident.
📋
Required section completeness
A template enforcement check ensures runbooks contain required sections (owner, last reviewed, rollback procedure). A runbook without a rollback procedure fails CI — the same way code without tests might fail CI in a TDD environment.

Where Docs as Code Works Best — and Where It Struggles

Docs as code produces the best results for documentation that's closely tied to code and changes when code changes: runbooks, API references, architecture decision records, and deployment procedures. It's most powerful in engineering-native organizations where engineers are already comfortable with Git workflows.

It struggles for documentation with a broader audience — onboarding guides, team wikis, cross-functional process docs — because requiring Git knowledge to edit documentation excludes non-engineers. Most mature implementations use docs as code for technical documentation (runbooks, architecture) and a separate wiki platform for broader team knowledge, with clear guidelines about what lives where.

The cultural requirement is the hardest part: teams need to believe that documentation changes are as real and important as code changes. If PRs that only touch documentation are treated as trivial — rubber-stamped without real review — the process produces the illusion of quality without the substance.

📌 Article Summary

What to Remember

Docs as code applies the same quality practices to documentation that engineering applies to code: version control (full change history), peer review (PR-based changes), automated checks (CI pipelines), and enforced ownership (CODEOWNERS). Each of these addresses a specific failure mode that wiki-based documentation doesn't handle.

The highest-value CI checks for documentation: broken links, stale review dates, references to deprecated services or commands, and required section completeness. These catch documentation failures before they reach engineers following runbooks under incident pressure.

Docs as code provides infrastructure for quality — it doesn't create documentation culture. The critical cultural requirement: documentation PRs are treated as real work requiring genuine review, not rubber-stamp overhead. Without that norm, the process adds workflow without adding quality.
