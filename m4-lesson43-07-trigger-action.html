Lesson 4.3 — No-Code & Low-Code Automation

What Is a Trigger and an Action? The Two Building Blocks of Every Automation

Every automation — from the simplest email notification to a complex multi-step workflow — is built from exactly two things: a trigger and one or more actions. This article makes those concepts concrete with examples across every tech role context you'll encounter.

The Grammar of Automation

Before you can build an automation, you need to be fluent in its grammar. Every automation is a sentence: When [trigger], do [action]. That's the complete structure. Everything you'll ever build in Zapier, Power Automate, Make, or any other automation tool is a version of that sentence.

The trigger is the "when." The action is the "do." Understanding each one deeply — what types exist, what information they carry, how they connect — is the difference between struggling with automation tools and moving through them confidently.

A trigger without an action is just a notification. An action without a trigger never runs. Together, they're a complete automated decision — executed reliably every time the condition is met.

🔵 Trigger
The "When"
A trigger is an event that starts a workflow. It watches for something to happen — in an app, on a schedule, or based on data — and fires the automation when the condition is met.
Something is created (new email, new row, new ticket)
Something changes (status updated, field modified)
A schedule is reached (every Monday at 9am)
A threshold is crossed (file size exceeds 80%, error rate above 2%)
A form is submitted or a button is clicked
🟢 Action
The "Do"
An action is what the automation does after the trigger fires. Actions use data from the trigger — plus data from previous actions — to perform a task in an app or service automatically.
Create something (a record, a file, a ticket, a calendar event)
Send something (an email, a Teams message, an SMS)
Update something (change a field, move a row, modify a record)
Retrieve something (look up a user, get a record, fetch data)
Notify someone (page on-call, post to Slack, send alert)

The Four Types of Triggers — With Real Examples

📥
Event-based trigger
Something happens in an app
The most common trigger type. The automation platform watches an app for a specific event — a new item created, a form submitted, a file uploaded — and fires when it happens. The platform polls the app periodically (every minute or faster) or receives a webhook notification instantly.
New ticket submitted in Zendesk · New row added to Google Sheets · Email received matching a filter · File uploaded to SharePoint folder · New user created in Active Directory
⏰
Schedule-based trigger
Time-based — runs on a cadence
The automation fires on a schedule rather than waiting for an event. Useful for recurring tasks that need to happen at a specific time regardless of whether anything "happened" — generating reports, sending reminders, checking for items that match a condition.
Every weekday at 8:45am · Every Monday at 4pm · First day of each month · Every 15 minutes · Every hour during business hours · Once a year on a specific date
📊
Data condition trigger
A value crosses a threshold
The automation monitors a metric or value and fires when it crosses a defined threshold — above or below a number, matching a specific value, or entering a specific state. Most common in monitoring and alerting workflows. Often combined with a schedule to poll the data source regularly.
Server CPU exceeds 85% · Disk usage above threshold · Ticket queue depth exceeds 50 items · Response time over 500ms · Stock level below reorder point · Failed login attempts exceed 10/hour
👆
Manual trigger
A human initiates it
The automation runs when a person explicitly triggers it — clicking a button, sending a command, or approving a request. Not fully automatic, but still eliminates repetitive steps once initiated. Useful for workflows that need human judgment to start but then run automatically from that point.
Click "Run report" button · Approve request in email · Send /command in Slack · Right-click a file → "Process with workflow" · Submit an internal tool form · Scan a QR code at a workstation

Beyond Simple: Conditions and Chains

Most real automations don't have just one trigger and one action. They have conditions — branches that take different actions depending on the trigger data — and chains — sequences of multiple actions that run in order. Here's a real example that shows both.

📋 Conditional workflow — IT access request with approval routing
📥
Trigger
An access request form is submitted. Data captured: employee name, system requested, business justification, manager's email, request urgency.
⚖️
Condition
Evaluate: is the requested system classified as "high sensitivity" (production database, HR system, financial records)?
IF system_sensitivity == "high" → BRANCH: requires_security_approval ELSE → BRANCH: manager_approval_only
🔒
Action — High sensitivity branch
Send approval request to manager AND security team simultaneously. Both must approve within 48 hours. If either rejects, deny the request and notify the employee with the reason.
✅
Action — Standard branch
Send approval request to manager only. Manager has 24 hours. If approved, immediately provision access. If no response in 24 hours, escalate to manager's manager.
🎉
Final action (both branches)
When approved: provision access, send confirmation to employee with login details, log the access grant in the audit trail, create a 90-day access review reminder.

This workflow has one trigger, one condition (with two branches), and multiple chained actions. Conditions let you handle different situations differently. Chains let you sequence multiple actions so each one uses the output of the previous step. This is still no-code or low-code — you build it visually in Power Automate or Make — but the logic is meaningfully complex.

📌 Article Summary

What to Remember

Every automation is "When [trigger], do [action]." The trigger is the event that starts the workflow — event-based (something happened), schedule-based (a time was reached), data condition (a threshold was crossed), or manual (a human initiated it). The action is what the automation does in response — create, send, update, retrieve, or notify.

Conditions let one trigger produce different responses depending on the data. Chains let multiple actions run sequentially from one trigger, each using data from previous steps. Most real automations combine trigger + condition + multiple chained actions — but the underlying grammar is always the same: when this, do that.

When you can't find the exact trigger you're looking for, think about what could serve as a proxy trigger for the condition you actually care about. "Check if unassigned tickets exist" isn't a trigger — "every morning at 8am" is, with a data lookup action immediately after. Framing what you want as a trigger + action pair almost always reveals the correct implementation, even when the natural language description doesn't map directly to a trigger type.
