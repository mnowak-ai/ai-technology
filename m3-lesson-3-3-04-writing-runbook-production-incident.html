Lesson 3.3 — Creating SOPs & Guides

How to Write a Runbook for a Production Incident: A Step-by-Step Guide

A runbook is the one document that has to work perfectly at the worst possible moment. Here's how to write one that actually does.

Runbooks Are Not SOPs in Disguise

A lot of teams write something they call a "runbook" that is really just a general SOP with a flashier name. This is a problem. A real production incident runbook is a fundamentally different kind of document — different structure, different detail level, different writing style, different update frequency.

The difference matters because of when and how it gets used: a runbook is read by someone who is stressed, possibly sleep-deprived, under time pressure, and responsible for restoring a service that real users are actively experiencing problems with. A vague step or an ambiguous command in that context isn't just unhelpful — it's dangerous.

Write every runbook step as if the person reading it will be doing so at 3am, on a phone, while their PagerDuty is still going off.

The Anatomy of a Good Runbook

We'll build a real runbook throughout this section — Runbook: AI Inference Service Degradation Response. Every section below is part of this same runbook.

1
Runbook Header
Read first

Before any steps, the runbook needs a header that tells the engineer: what alert this is for, what service it covers, who owns it, and when it was last updated. This isn't optional — a stale runbook from 8 months ago should be immediately obvious.

📄 Example Header
RUNBOOK: AI Inference Service — Degradation Response
ID: RB-ML-007
Triggered by: PagerDuty alert "ai-inference-high-latency-prod"
Service: inference-api (production)
Owner: ML Platform On-Call
Last updated: 2024-11-15 | Next review: 2025-02-15
Version: 2.3
2
Alert Context
What triggered this

What does this alert mean? What does it measure? What threshold triggered it? The engineer reading this may not be the team member who set up the alert — they need to understand what they're looking at before they start diagnosing.

📄 Example
Alert condition: p99 request latency > 2,000ms for 5 consecutive minutes
Normal baseline: p99 < 400ms
What this means: The AI inference service is returning responses significantly slower than normal. Users experience slow or timing-out predictions. Dependent services (recommendations-api, fraud-detection) may also degrade within 10–15 minutes if not resolved.
3
Immediate Actions (First 5 Minutes)
Do these first

The first steps of a runbook are the most critical. These are the actions that stop the bleeding — acknowledging the alert, communicating to the team, and beginning the initial triage. Keep this section to 3–5 steps maximum. These should be fast, clear, and non-diagnostic.

📄 Example
1. Acknowledge the PagerDuty alert (stops escalation timer)
2. Post in #ai-oncall: "🔴 Investigating ai-inference-high-latency-prod — [your name] — [timestamp]"
3. Open Grafana: AI Inference — Production dashboard
4. Note current p99 latency and error rate. Screenshot and paste in #ai-oncall.
5. Check if a deployment occurred in the last 60 minutes: kubectl rollout history deployment/inference-api -n ai-prod
4
Diagnostic Decision Tree
Figure out what's wrong

This is the most important and most often missing section of a runbook. The diagnostic section tells the engineer what to check, in what order, and what each result means. It should be a decision tree — not a flat list of "check X, check Y, check Z." Real incidents branch. Good runbooks branch too.

📄 Example — Decision Tree Format
DIAGNOSTIC PATH:

→ Check GPU utilization in Grafana (GPU Utilization panel)
   If GPU > 90% for 5+ minutes: → Go to Section A (GPU Saturation)
   If GPU is normal (<70%): → Check recent deployments

→ Check deployment history:
   kubectl rollout history deployment/inference-api -n ai-prod
   If new deployment in last 60 min: → Go to Section B (Deployment Rollback)
   If no recent deployment: → Go to Section C (Model Health Check)
⚠️ A flat list of diagnostic steps without branching forces the engineer to read through steps that don't apply to their situation — wasting time at exactly the wrong moment.
5
Resolution Sections (One Per Scenario)
Fix the thing

Each branch in the diagnostic tree leads to a resolution section. Each resolution section gives exact steps, exact commands, exact success criteria. Don't merge scenarios into one section to save space — under pressure, engineers need to find the right section immediately.

📄 Example — Section B: Deployment Rollback
SECTION B: Recent Deployment — Initiate Rollback

B1. Confirm which deployment version is current:
    kubectl describe deployment/inference-api -n ai-prod | grep Image

B2. Roll back to previous version:
    kubectl rollout undo deployment/inference-api -n ai-prod

B3. Watch rollout status:
    kubectl rollout status deployment/inference-api -n ai-prod
    Wait until: "deployment successfully rolled out"

B4. Verify recovery: Check Grafana. Latency should return to <400ms p99 within 3 minutes.
    If latency does not recover within 5 minutes of rollback: → Go to Section C.

B5. Post in #ai-oncall: "Rollback complete. Monitoring recovery. Previous version: [version]."
6
Escalation Triggers
Know when to get help

An escalation section is not a failure condition — it's a decision support tool. It tells the engineer: "If X, Y, or Z is true after following this runbook, here is who to call and how." Be specific about conditions, contacts, and method.

📄 Example
ESCALATE TO ML PLATFORM ON-CALL IF:
• Root cause is not identified within 30 minutes
• Rollback does not restore service within 10 minutes of completion
• Model accuracy has degraded (check accuracy monitor in Grafana)

HOW: Page PagerDuty policy "ml-platform-oncall"
Post in #ml-platform: include Grafana link, current metrics, steps already taken

ESCALATE TO P1 IF:
• Error rate exceeds 20% for 10+ consecutive minutes
• More than 2 downstream services affected
HOW: Post in #incident-response. Ping @incident-commander.
7
Post-Incident Notes Section
After resolution

Every runbook should have a notes field at the bottom. After an incident is resolved, the engineer who used the runbook fills in: what they found, what worked, what didn't, and whether the runbook was accurate. This is how runbooks improve over time — from real incident data, not theory.

📄 Example
POST-INCIDENT NOTES:
Date: _____ | Incident ID: _____ | Filled in by: _____

Root cause found: _____
Steps that were unclear or incorrect: _____
Steps that were missing: _____
Did this runbook resolve the incident? [ ] Yes [ ] No [ ] Partially
Suggested improvements: _____

What Makes Runbooks Fail Under Pressure

❌
Vague success criteria. "Confirm the service is healthy" — what does healthy mean? What metric? What threshold? What timeframe? Engineers have restarted services and assumed everything was fine because they didn't know what number they were looking for.
❌
No rollback step. Every action in a runbook that changes system state needs a corresponding rollback. If you tell an engineer to change a configuration, tell them how to undo it if it makes things worse.
❌
Flat lists instead of decision trees. Real incidents have multiple possible causes. A runbook that lists every possible check in a flat numbered list forces the engineer to work through irrelevant steps — losing minutes they don't have.
❌
Dead links and stale dashboards. Grafana dashboards get renamed. Confluence pages get moved. A runbook with a dead link stops an engineer cold while they hunt for the right URL. Dashboard links must be checked at every review.

📌 Article Summary

What to Remember

A production incident runbook has seven parts: Header, Alert Context, Immediate Actions, Diagnostic Decision Tree, Resolution Sections (one per scenario), Escalation Triggers, and Post-Incident Notes. Each section must be specific enough to follow at 3am with no additional context.

The diagnostic section must be a decision tree, not a flat list. Incidents branch — a flat list forces engineers to read through irrelevant steps. Every diagnostic check should answer: "If I see X, go here. If I see Y, go there."

Four common runbook failures: vague success criteria, no rollback steps, flat lists instead of decision trees, and dead links. Each one of these has extended real incidents. Check for all four at every runbook review.
