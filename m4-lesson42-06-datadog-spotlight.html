How Datadog AI Monitors Infrastructure in Real Time

The video's Datadog case study covered one slide. This article gives you the full picture: what Datadog actually monitors, how its AI features work, what the workflow looks like when an alert fires, and exactly where human judgment enters the process.

🐶
Datadog
Cloud infrastructure monitoring, APM, log management, and AI-powered anomaly detection. Used by over 27,000 organizations to monitor billions of metrics per second across distributed infrastructure.
datadoghq.com/solutions/ai

Datadog's job is to make the invisible visible. Modern tech infrastructure runs across dozens of servers, cloud services, databases, and network components simultaneously — no human can watch all of them at once. Datadog centralizes all of that observability data into one place, and its AI layer (called Watchdog) continuously scans it for patterns and anomalies that no human team could detect at that speed and scale.

What Datadog Actually Monitors

Datadog collects four major categories of observability data. Understanding what's in each category helps you understand what kinds of problems Datadog can and can't detect.

📊
Infrastructure Metrics
Quantitative measurements of server and system health, collected every few seconds from every monitored host. This is the "vital signs" data — the continuous health checkup running 24/7 on every piece of infrastructure.
CPU usage %
Memory used/free
Disk I/O
Network bytes/sec
Process count
🌐
Application Performance (APM)
Metrics that measure how well the software running on the infrastructure is actually performing — from the user's perspective. APM tracks every request flowing through an application and how long each step takes.
Request latency
Error rate (4xx/5xx)
Throughput (req/sec)
Apdex score
Database query time
📋
Log Management
All log output from all monitored services, centralized in one searchable index. Instead of SSH-ing into individual servers to read logs, engineers can search across all logs from all services simultaneously and see correlated events across the entire system.
Error messages
Access logs
Deployment events
Application warnings
Security events
🔔
Synthetic Monitoring
Automated tests that simulate real user actions on a schedule — clicking through a checkout flow, logging in, loading a dashboard — and alert if the experience degrades or fails. This catches user-facing problems before real users report them.
Uptime checks
Browser tests
API endpoint tests
SSL cert expiry

Watchdog: Datadog's AI Anomaly Detection Engine

Watchdog is the AI component that continuously scans all of the above data for patterns that deviate from learned baselines. It doesn't wait for a human to define threshold rules — it learns what "normal" looks like for each metric on each service, and flags deviations automatically.

📈
Automated Metric Anomaly Detection
Watchdog learns baseline behavior for every monitored metric — accounting for time-of-day patterns, day-of-week variation, and seasonal trends. When a metric deviates significantly from its learned baseline, Watchdog surfaces it automatically. Unlike static threshold alerts ("fire when CPU exceeds 90%"), Watchdog adapts to each service's unique normal behavior: a 70% CPU on a database server might be normal during nightly batch jobs, while 70% on a usually-idle cache server is anomalous.
👤 Human role: After Watchdog flags an anomaly, a human investigates whether it represents a real problem or a false positive. Watchdog shows the deviation — it doesn't determine the cause or severity.
🔗
Correlated Incident Detection
When Watchdog detects anomalies across multiple related metrics simultaneously — say, elevated error rates, increased response time, and rising database query latency all happening at the same time on related services — it groups them into a single correlated incident rather than firing separate alerts for each. This dramatically reduces alert noise and gives engineers a single coherent view of a problem rather than a flood of individual metric alerts.
👤 Human role: Review the correlated incident to understand the scope. Which services are affected? Which is the root cause and which are downstream effects? Correlated detection shows what's happening together — it doesn't tell you what caused what.
🔍
Log Pattern Analysis
Watchdog scans log streams to identify new error patterns that haven't appeared before, or existing error patterns that are suddenly appearing at a higher rate than usual. Rather than requiring engineers to manually grep through millions of log lines, it surfaces the patterns that matter. It can also correlate log anomalies with metric anomalies — showing, for example, that an error message that started appearing in the logs 10 minutes ago correlates with the latency spike in the metrics.
👤 Human role: Read the actual log lines Watchdog surfaced to understand the nature of the error. Is this a new error or a regression? Does the error message point to a specific component, query, or code path?

Watchdog detected three anomalies, grouped them into one incident (rather than three separate alerts), and identified a likely causal sequence: the database latency anomaly appeared 4 minutes before the payment service errors — suggesting the database is the root cause, not the application.
👤
Human — On-call engineer (paged)
Receive alert, open Datadog dashboard, assess scope

The on-call engineer receives a PagerDuty page at 14:37. They open the Datadog incident view on their laptop. They see: 847 users per minute are receiving 503 errors on the payment service. The correlated incident shows database query latency is 70× above normal. They check the deployment log — no deployments in the past 2 hours. They check the database connection pool metrics — the connection pool is at 98% capacity, with threads queueing. This points to a connection pool exhaustion, not a query performance regression.
👤
Human — On-call engineer
Search correlated logs to identify the cause
Click to see what the logs reveal →
Log pattern surfaced by Watchdog: First occurrence: 14:28:14 — new pattern ERROR [payment-db] Connection pool exhausted pool_size: 100 | active: 100 | waiting: 247 query: SELECT * FROM transactions WHERE user_id IN (...) — 47,821 user IDs Frequency: 847 occurrences in last 5 minutes Prior 30 days: 0 occurrences
Watchdog surfaced a log pattern that appeared for the first time at 14:28 — 9 minutes before it was flagged (it didn't trigger the alert threshold until it became sustained). The root cause: a new background job is running a query with 47,821 user IDs in a single IN() clause — an unbounded query that saturates all database connections and prevents the payment service from getting a connection.
👤
Human — Engineering team
Decide on mitigation and implement fix
Click to see the human decision →
The engineer identifies that the background job was deployed 3 days ago but only now ran against the full user table for the first time (it was tested against a small dataset). Immediate mitigation: kill the background job, restart the payment service connection pool. Total downtime: 14 minutes. Root cause fix: the query is refactored to batch users in groups of 1,000 rather than all at once. A query timeout limit is added to the background job configuration to prevent future runaway queries from exhausting the connection pool.
🤖
AI — Watchdog
Confirm recovery — metrics return to normal
Click to see the all-clear →
✅ WATCHDOG INCIDENT RESOLVED — 14:51 UTC payment-service error rate: 0.08% (normal) payment-db query latency: 52ms (normal) payment-service 503 rate: 0/min (normal) Incident duration: 14 minutes Users impacted: ~11,900 failed payment attempts
Once the connection pool is restarted, all three anomaly metrics return to their expected ranges within 90 seconds. Watchdog automatically resolves the incident and sends the resolution notification. The engineer documents the postmortem: root cause, timeline, mitigation, and the code change that prevents recurrence.

What Datadog AI Can't Do — The Honest Version

🧠
Can't explain why
Watchdog knew that database latency was 70× above normal. It couldn't tell you that an unbounded IN() clause with 47,821 IDs was the cause. Root cause analysis required reading the actual log lines.
📅
Can't know what changed
Watchdog detected an anomaly onset but didn't know a background job was deployed 3 days ago and ran against the full table for the first time today. Deployment history context is human-supplied.
⚖️
Can't decide the fix
Once the cause was identified, the engineer had to decide: kill the job immediately vs. let it finish vs. throttle it. That tradeoff — speed vs. data completeness — requires business judgment.
🔮
Can miss slow-building problems
If the unbounded query had run daily but grown slowly from 1,000 users to 47,821 over months, each day's baseline would have adjusted. The trend might never have triggered an alert until it was too late.
🧠 Quick Check
Datadog's Watchdog flagged a high-severity correlated incident: both the API gateway error rate and the database query latency are anomalous simultaneously. A junior engineer sees the incident and immediately scales up the number of API gateway servers (adding more compute capacity). The incident continues. 📌 Article Summary

What to Remember

Datadog monitors four categories of infrastructure observability: infrastructure metrics (server vital signs), application performance metrics (request speed and error rates), log management (all log output searchable in one place), and synthetic monitoring (automated user journey tests). Its AI engine, Watchdog, scans all of this continuously for anomalies — at a scale no human team could match manually.

Watchdog's most important capability is correlated incident detection: grouping multiple simultaneous anomalies across related services into one incident, and identifying which anomaly appeared first (the likely cause) vs. which appeared later (the downstream effects). This dramatically reduces alert noise and gives engineers a single coherent view of a problem.

Datadog finds the anomaly. Humans find the cause, decide the fix, and implement the solution. The 14-minute incident in this article was resolved because a human read the actual log lines (surfaced by Watchdog) and identified the specific unbounded query — something no automated system could have diagnosed from metrics alone. AI and human work are sequential, not competing.
