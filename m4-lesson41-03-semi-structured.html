Lesson 4.1 — Data Literacy for Tech Roles

Semi-Structured Data: The Type Everyone Forgets (And Why JSON and XML Are Everywhere in Tech)

The video's quiz tested you on semi-structured data — but never explained it. This article fills that gap completely. By the end, you'll recognize JSON and XML in the wild and understand exactly why they're the backbone of modern AI integrations.

What Makes Data "Semi-Structured"?

Fully structured data (like a database table) has a rigid schema: every record has the same fields, in the same order, with the same data types. Fully unstructured data (like an email) has no schema at all — it's just free text.

Semi-structured data sits between these two. It has self-describing tags or markers that give it some organization, but it doesn't fit into rows and columns. Instead of a rigid table, it's more like a labeled container: the data uses labels to identify what each piece of information is, but the structure can vary from record to record, and values can be nested inside other values.

The most important semi-structured formats in tech are JSON and XML. If you work in any tech role, you will encounter both — in API responses, in configuration files, in log processing pipelines, and in virtually every system that needs to send data between applications.

Every time a mobile app asks a server for data and the server sends a response — the conversation almost always happens in JSON. Every AI API integration you'll ever encounter uses it. Semi-structured data isn't obscure. It's the water tech runs in.

JSON and XML — Side by Side

The best way to understand these formats is to see them. Both formats below represent the same support ticket record. Click each tab to explore the format and see what makes it "semi-structured."

The actual JSON data
{ "ticket_id": "TKT-8821", "created_at": "2024-11-14T09:14:02Z", "priority": "high", "category": "vpn_access", "status": "open", "submitter": { "user_id": "usr_4471", "name": "Jessica Chen", "department": "Sales" }, "description": "Can't connect to VPN. Auth failed error. Board presentation in 20 min.", "tags": [ "vpn", "urgent", "auth" ], "sla_hours": 4, "assigned_to": null }
What makes this semi-structured
Key-value pairs
Every piece of data has a label (the "key" in quotes) followed by its value. A machine can extract any field by name: ticket_id, priority, status. No need to know the column number — the label is self-describing.
Nested objects
The "submitter" field isn't a simple value — it contains its own set of key-value pairs (user_id, name, department). This nesting is something a flat database table can't represent cleanly. JSON handles complex relationships naturally.
Arrays
The "tags" field is a list of values (an array). One ticket can have multiple tags. In a flat database, this would require a separate table and a join query. In JSON, it's just a list inside the same record.
Flexible structure
Different tickets can have different fields. One ticket might have an "escalation_reason" field; another might not. JSON handles this — a database table would need every column defined in advance.
📡 Where you see JSON in tech: API responses (every time an app talks to a server), configuration files (how AI tools are configured), webhook payloads (how automated systems notify each other of events), log aggregation pipelines, and the input/output format of virtually every modern AI API including the tools used throughout this course.
The same ticket in XML
<ticket> <ticket_id>TKT-8821</ticket_id> <created_at>2024-11-14T09:14:02Z</created_at> <priority>high</priority> <category>vpn_access</category> <status>open</status> <submitter user_id="usr_4471"> <name>Jessica Chen</name> <department>Sales</department> </submitter> <description> Can't connect to VPN. Auth failed error. Board presentation in 20 min. </description> <tags> <tag>vpn</tag> <tag>urgent</tag> <tag>auth</tag> </tags> <sla_hours>4</sla_hours> </ticket>
What makes XML different from JSON
Opening and closing tags
Instead of key-value pairs, XML uses opening tags (<priority>) and closing tags (</priority>) to mark each field. The content between the tags is the value. If you've seen HTML, this syntax looks familiar — they're related formats.
Attributes inside tags
Tags can contain attributes (user_id="usr_4471" inside the <submitter> tag). This is different from JSON — it's a way of embedding metadata about an element directly in its tag.
More verbose than JSON
XML requires both an opening and closing tag for every field, making it significantly longer than the equivalent JSON. For this reason, modern systems tend to prefer JSON — but XML remains widespread in older systems and enterprise integrations.
Still self-describing
Like JSON, XML is self-describing — the tags tell you what each piece of data is. A machine can extract the priority value by finding the content between <priority> and </priority> tags, without needing to know a column number.
📡 Where you see XML in tech: Enterprise software integrations (especially older systems), configuration files for many applications, RSS/Atom feeds, SOAP web services (an older API standard still common in finance and healthcare), Microsoft Office document formats (.docx, .xlsx are actually XML inside a ZIP file), and network device configuration exports.

Where Semi-Structured Data Lives in Real Tech Workflows

AI Tool Integrations
API requests and responses
When an automation tool sends a ticket to an AI classification service and gets a category back, both the request and response are JSON. Every modern AI API — including the ones powering tools in this course — speaks JSON.
{"ticket": "VPN not working", "category": "network_access", "confidence": 0.94}
Log Processing
Structured log formats (JSON logs)
Modern applications increasingly write logs as JSON instead of plain text. This makes them machine-parseable without custom text processing — AI monitoring tools can read JSON logs directly and extract any field they need.
{"ts":"2024-11-14T14:42:08Z", "level":"ERROR", "msg":"auth failed", "user":"usr_4471"}
Automation Workflows
Webhook payloads
When a system wants to notify another system that something happened (a new ticket was created, a payment was processed, a deployment completed), it sends a JSON payload to a webhook URL. This is how Power Automate, Zapier, and most no-code tools receive triggers.
{"event": "ticket.created", "ticket_id": "TKT-8821", "priority": "high", "timestamp": "2024-11-14..."}
Configuration
Application configuration files
Most software is configured through JSON or XML files — telling the application what database to connect to, what thresholds to use for alerts, what features to enable. Tech workers editing these files are working with semi-structured data directly.
{"alert_threshold": 0.05, "auto_escalate": true, "sla_hours": {"high":4, "low":24}}
Data Exchange
System-to-system data transfers
When an HR system exports employee records to an IT provisioning system, or when a CRM exports customer data to a support tool, the transfer format is almost always JSON or XML. Semi-structured data is the universal language of system integration.
<employee id="4471"> <name>Jessica Chen</name> <access_level>standard</access_level> </employee>
Mobile & Web Apps
Every app on your phone uses JSON
When you open an app and it loads your data — your messages, your account info, your settings — the server sends that data as JSON. When you submit a form or make a request, your app sends JSON back. Every modern app conversation is in semi-structured data.
{"user": {"id": "usr_4471", "name": "Jessica Chen", "notifications": 3, "theme": "dark"}}

📌 Article Summary

What to Remember

Semi-structured data has self-describing labels that give it organization, but doesn't fit into rows and columns. JSON uses key-value pairs and can nest objects and arrays. XML uses opening and closing tags. Both formats are self-describing: a machine can extract any field by name without knowing column positions.

JSON is everywhere in modern tech. Every API response, every automation webhook, every AI tool integration, every mobile app data exchange uses JSON. You don't need to write JSON code in most tech roles — but you need to be able to read it, understand its structure, and recognize when a field is missing or has changed.

Semi-structured data is the connective tissue of AI integrations. When an automation workflow sends data to an AI API and receives a classification result, both ends of that conversation are JSON. Understanding this format is what lets you troubleshoot when integrations break — recognizing that "field not found" errors mean the API's structure changed, not that the routing logic is wrong.
