Lesson 3.3 — Creating SOPs & Guides

Keeping SOPs and Runbooks Current: Building a Review Cycle That Actually Gets Done

Every team intends to keep their SOPs up to date. Almost none of them do. Here's what actually works — and why the calendar-based review cycle almost always fails.

The Problem With "We Review These Every Quarter"

Most teams that have SOPs and runbooks have a "review process" that looks like this: someone set a recurring calendar invite six months ago for a quarterly review. That invite gets declined, postponed, or ignored. When it does happen, the reviewer skims the document, doesn't find anything obviously wrong, marks it "reviewed," and updates the date.

Six months later, during an incident, an engineer follows the runbook — and the dashboard URL returns a 404. The Kubernetes namespace it references no longer exists. The command uses a deprecated flag. The runbook was "reviewed" two months ago and it's completely wrong.

A document with a "last reviewed" date is not the same as a document that's accurate. The review process matters as much as the review schedule.

Why Calendar-Only Review Cycles Fail

❌ Problem 1: Reviews Happen Whether or Not Anything Changed
Quarterly reviews don't care if nothing changed or if everything changed. A runbook for a system that was completely re-architected two months ago might not be reviewed until the calendar hits — by which point it's been used and caused confusion.
❌ Problem 2: No One Owns the Review
A team-level calendar invite doesn't mean a specific person is responsible for a specific document. "The team" reviews it — meaning whoever happens to be available, skimming it, looking for nothing in particular.
❌ Problem 3: No One Knows What "Current" Means
Reviewers check for obvious errors but don't verify commands, test links, or compare steps against how the system actually works today. The review is a document-level check, not a ground-truth check.
❌ Problem 4: Incidents That Should Trigger Reviews Don't
Every incident where a runbook was used and found lacking is a perfect opportunity to improve it — immediately, while the memory is fresh. These trigger-based updates almost never happen because the incident team is exhausted, the postmortem is backlogged, and updating the runbook falls off the list.

What Actually Works: Trigger-Based Reviews

The most reliable way to keep SOPs and runbooks current is to tie review triggers to real events — not just the calendar. Calendar-based reviews are a fallback. Event-based triggers are the primary mechanism.

Here are the five triggers that should automatically kick off a document review:

🔴
Trigger 1: Any incident that involved this document
If a runbook was used during an incident — successfully or not — it gets reviewed immediately after the incident is resolved. This is the single most valuable review trigger because it catches real-world gaps while the memory is fresh.
Example: An on-call engineer followed a runbook for a Redis cache degradation but found the monitoring URL had changed. That gets fixed the day after the incident — not in three months.
🟠
Trigger 2: Any infrastructure or tooling change that touches this procedure
When a system is migrated, a tool is upgraded, or a namespace is renamed, every SOP and runbook that references that system gets flagged for review before the change goes live.
Example: A Kubernetes cluster migration from v1.25 to v1.28 should trigger review of every runbook with kubectl commands — some flags and behaviors changed between versions.
🟡
Trigger 3: Team membership changes
When the engineer who wrote the runbook leaves the team, or when a new engineer joins and is expected to use it, the document needs a review — ideally as part of knowledge transfer and onboarding.
Example: A senior engineer who owns the AI model deployment SOP leaves. During their offboarding, they review and update every procedure they own. The new owner validates it on their second week.
🟢
Trigger 4: Quarterly calendar review (as a backstop)
Calendar reviews still happen — but as a catch-all for documents that haven't been triggered by any of the above events. They're not the primary mechanism; they're the safety net.
The quarterly review checklist: verify every link works, run each command in staging to confirm it still works, check that every role and contact is current, verify the escalation policy name is still active in PagerDuty.
🤖
Trigger 5: AI-assisted staleness detection
Emerging AI tools can scan documentation and flag potential staleness: references to deprecated APIs, tool versions that no longer match current ones, or internal URLs that return errors. Teams using tools like Confluence AI or custom documentation pipelines can automate this scan on a weekly basis.
Example: An AI documentation scanner finds that a runbook references "kubectl --generator=deployment/v1beta1" — a flag removed in Kubernetes 1.18. It flags the runbook for immediate review.

Ownership: The Most Ignored Part of SOP Maintenance

A document with no single named owner doesn't get updated. "The team owns it" is not ownership — it's diffused responsibility that no one actually acts on.

Every SOP and runbook needs one person named as owner. Not a team. A person. That person is accountable for making sure the document is reviewed on trigger events and that review tasks get completed.

📌 Article Summary

What to Remember

Calendar-only review cycles fail for four reasons: they're not tied to real changes, no one person owns the review, reviewers don't know what "verified" means, and incident-based updates almost never happen. A good maintenance system combines trigger-based reviews with a calendar backstop.

The five review triggers: any incident that used the document, any infrastructure change that touches it, team membership changes, the calendar backstop, and AI-assisted staleness detection. Infrastructure change and incident triggers are the most important — they catch the gaps that matter most.

A real review verifies: every command still works (test it), every link is live (click it), every contact and escalation path is current (confirm it), and every system reference matches the actual environment (check it). Reading and not finding obvious errors is not a verification.
