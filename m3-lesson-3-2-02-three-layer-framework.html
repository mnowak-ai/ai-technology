Lesson 3.2 — Summarizing Technical Information

The Three-Layer Summary Framework: Executive, Operational, and Technical

One production outage. Three completely different summaries. Here's the framework — and the exact documents — that distinguish engineers who communicate well from those who don't.

Why Three Layers?

Any significant technical event — an outage, a security incident, a model degradation, a failed deployment — generates information that multiple audiences need to act on. Those audiences have different jobs, different vocabulary, and different definitions of "what I need to know."

The three-layer framework gives each audience a document built specifically for their job — not a single document that tries to serve everyone and ends up serving no one well.

The executive layer enables decisions. The operational layer enables response. The technical layer enables fixes. Knowing which layer you're writing prevents you from putting the wrong information in the wrong document.

Technical Summary

POSTMORTEM SUMMARY — P1-2024-1115-PAYMENT Duration: 6h 14m | Severity: P1 | Status: Resolved ROOT CAUSE Memory leak in payment-service v4.1.2, introduced in PR #2341 (merged Nov 13, deployed Nov 14 22:40 UTC). The leak occurs in the Stripe webhook retry handler (stripe_webhook.py:retry_handler()). Failed webhook callbacks accumulate in memory without garbage collection when retry queue depth exceeds 500 items. Under normal traffic this threshold is not reached. Under the Black Friday pre-sale traffic spike on Nov 15 AM, retry volume crossed threshold, memory reached the 2GB pod limit, pod OOMKilled, and did not recover cleanly on restart due to uncleaned retry state. CONTRIBUTING FACTOR PagerDuty alert "payment-memory-high" (fires at 1.5GB) was suppressed by a misconfigured rule introduced in the Nov 8 monitoring update. Rule: "suppress if pod restart count < 2 in 30 min." Pod restart counter was not incrementing correctly post-Nov 8 update, so alert never fired. Detection was delayed by ~47 minutes — incident was ultimately caught via customer support ticket volume. 📌 Article Summary

What to Remember

The three-layer framework maps to three jobs: executive (enable decisions), operational (enable response), technical (enable fixes). Each layer uses different vocabulary, different depth, and different framing — because each reader needs something different to do their job correctly.

What never changes across layers: the facts. The timeline is the same. The resolution status is the same. Nothing is hidden in one layer that would change a decision in another. Adapting the framing is appropriate — concealing material information is not.

Most engineers default to the technical layer for all audiences — it's what they know best. The career-differentiating skill is being able to shift between layers deliberately: writing a clean executive summary, an actionable operational brief, and a rigorous technical postmortem from the same set of facts.
