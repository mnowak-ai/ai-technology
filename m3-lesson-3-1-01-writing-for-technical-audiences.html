Lesson 3.1 — Clear Technical Communication

Writing for Technical Audiences: How to Be Clear Without Being Condescending

Your readers are smart. That's why vague writing is so dangerous — they'll act on whatever you wrote, even when it's unclear. Here's how to be precise without talking down to anyone.

The Trap Most Technical Writers Fall Into

When you write for other engineers, you face two opposite problems. Write too vaguely and people misread your instructions. Write too thoroughly and people feel like you're explaining obvious things. Both create friction. Both create risk.

The goal of clear technical writing isn't to dumb things down. It's to remove every possible reason someone could misread what you wrote. That's a precision problem, not a complexity problem.

Here's the core insight: technical readers don't slow down when they're confused. They fill in the gaps with their own assumptions and keep moving. That's how misconfigured deployments happen, how the wrong server gets restarted at 3am, and how an AI-generated alert summary causes someone to close the wrong ticket.

Clarity is not the opposite of technical depth. It's what technical depth looks like when it's written for the reader, not the writer.

Six Rewrites You Can Learn From

Let's look at real examples from IT work. For each one, we'll see the unclear version, the clearer rewrite, and — most importantly — why the original failed.

Rewrite #1 — Incident Report
SRE / On-Call
❌ Unclear
The system experienced degraded performance due to resource issues. Engineers were notified and the situation was addressed. Monitoring has been restored.
✅ Clear
At 02:14 UTC, the inference API returned HTTP 503 errors for 11% of requests due to memory exhaustion on two of three pod replicas. The on-call SRE restarted the affected pods at 02:31 UTC. Error rate returned to baseline (<0.1%) by 02:38 UTC. Root cause: a memory leak in the batch scoring job introduced in v2.3.1. That job has been disabled pending a fix. Monitoring alert thresholds remain active.
Why the original failed
The original describes that something happened, not what happened or whether it's really fixed. "Resource issues" could mean CPU, memory, disk, or network. "Addressed" doesn't say how. "Monitoring restored" implies monitoring was gone — but what exactly is being monitored now? The rewrite gives future engineers enough information to act: the specific failure mode, the fix, the timeline, and the open risk (the disabled job).
Rewrite #2 — Jira Ticket
Engineering / Jira
❌ Unclear
Bug: the model predictions are wrong sometimes. Seen it multiple times this week. Seems like a data thing. Assigning to ML team.
✅ Clear
Model returns confidence scores above 0.95 for inputs that contain null values in the `user_region` field. Expected behavior: inputs with null `user_region` should be flagged as low-confidence (<0.4) and routed to the fallback rule engine. Observed 14 times this week (Mon–Thu). Reproduces consistently: send a scoring request with `"user_region": null` to /api/v2/predict. See attached logs from 2025-11-12 11:40 UTC. Suspected cause: null handling change in the feature pipeline (PR #841, merged Nov 9). Assigning to ML team.
Why the original failed
"Wrong sometimes" is not actionable. The ML team would need to ask four follow-up questions before they could even start debugging. "Seems like a data thing" is a hypothesis, not evidence. The rewrite includes: the exact input condition, the expected vs. observed behavior, reproduction steps, frequency, timestamps, and a suspected cause with a pointer to relevant code. An engineer can start working on this ticket without a single clarifying question.
Rewrite #3 — AI-Generated Alert Summary
AI Tool Output
❌ AI Draft (needs review)
Alert triggered at 09:42. Anomaly detected in production workload. Elevated error rates observed. Team should investigate the affected service and restore normal operations as soon as possible.
✅ After SRE Review
🔴 P1 Alert — 09:42 UTC | payment-service in us-east-1

Error rate: 23% (threshold: 1%) | Affected endpoint: POST /api/checkout/confirm

Likely cause: upstream timeout from stripe-gateway (latency p99 = 8.4s, SLA = 2s). This is NOT a code regression — no recent deployments. Probable external dependency issue.

Immediate action: Check Stripe status page. If Stripe confirms degradation, activate fallback: enable async payment queue (runbook §4.2). If Stripe shows healthy, escalate to payments-eng channel.
Why the AI draft failed — and what the SRE added
The AI-generated alert tells you something is wrong. The SRE rewrite tells you what, where, why, and what to do. The SRE added: the specific service and region, the actual error rate vs. threshold, the suspected external cause (not a code bug), and a concrete decision tree. Under pressure at 3am, the difference between these two versions is the difference between a 10-minute recovery and a 2-hour investigation.
Rewrite #4 — PR Description
Code Review
❌ Unclear
Updated the caching layer. Should fix the performance issues we've been seeing. Tested locally.
✅ Clear
Replaces Redis TTL-based cache with LRU eviction policy for the model feature store (affects `feature_cache.py` only). Change reduces cache churn by ~40% in our load test (see benchmark in /tests/perf/cache_lru_20251113.txt). Why now: TTL approach was causing cold-start latency spikes during traffic bursts — see issue #302. Risk: LRU size is currently set to 5,000 entries (same as old TTL cache). If feature space grows beyond this, we'll evict useful entries. Recommend monitoring `cache_eviction_rate` metric in Datadog after deploy. Rollback: revert this PR and redeploy — no DB migration needed.
Why the original failed
"Updated the caching layer" gives reviewers nothing to evaluate. The clear version tells reviewers what changed, what problem it solves, the trade-off introduced (LRU eviction risk), what to watch after deploy, and how to roll back safely. Good PR descriptions aren't just courtesy — they're documentation that other engineers will reference for months.
Rewrite #5 — Runbook Step
Operations
❌ Unclear
If the model is not responding, restart the service and check that it comes back up correctly.
✅ Clear
If the model service health check returns non-200 for 3 consecutive checks (30-second interval):

1. Run: kubectl rollout restart deployment/inference-api -n ml-prod
2. Wait 90 seconds, then run: kubectl get pods -n ml-prod -l app=inference-api
3. Confirm all pods show STATUS=Running and READY=1/1
4. Run health check manually: curl https://inference.internal/health
— Expected response: {"status":"ok","model":"v4.2","uptime":<seconds>}
5. If any pod is in CrashLoopBackOff, do NOT retry — escalate to ml-oncall (PagerDuty: ml-infra-p1).
Why the original failed
"Check that it comes back up correctly" is not a procedure — it's a hope. The unclear version requires the engineer to know what "not responding" means, what command to run, what a successful restart looks like, and what to do if it doesn't work. The clear version specifies the trigger condition, the exact commands, the expected output, and the escalation path. Under incident conditions, ambiguity in runbooks causes engineers to improvise — which is exactly when mistakes happen.
Rewrite #6 — Deployment Email to the Team
Team Communication
❌ Unclear
Heads up — we're deploying some changes this afternoon. There may be some brief disruption. Let us know if you see any issues.
✅ Clear
Deployment notice — Friday Nov 15, 4:00–4:30 PM UTC

What's changing: updating the recommendation engine from v3.1 to v3.2 (adds support for multi-language user profiles).

Expected impact: the /api/recommend endpoint will be unavailable for ~3 minutes during pod rotation (approx. 4:12 PM UTC). All other endpoints remain available.

What to watch for: if you see recommendation response times above 500ms after 4:30 PM, ping #deployments. No action needed from most teams — this only affects the consumer app's "For You" section.

Rollback plan is ready if needed. Will confirm when deploy is complete.
Why the original failed
"Brief disruption" means different things to different teams. "Some changes" doesn't tell dependent teams whether they're affected. "Let us know if you see issues" doesn't say what an issue looks like. The clear version tells each team exactly whether they're affected, what the window is, what to watch for, and who to contact. Teams that don't need to act can confidently ignore it. Teams that do need to monitor have everything they need.

The Underlying Principle: Write for What Happens Next

Every piece of technical writing you produce will be read by someone who needs to do something after reading it. They might restart a service, approve a change, escalate an alert, or route a ticket. If your writing doesn't give them what they need to take the right next action, you've created risk.

Ask yourself this before you finish any technical document:

🎯
What action does the reader need to take?
Every runbook step, ticket, or alert summary exists because someone needs to act. Does your writing tell them exactly what that action is — or does it leave room for interpretation?
🧩
What information does the reader need to take it correctly?
Not just that the problem exists, but: which specific system, what exact behavior, which version, what the expected result is, what to do if it doesn't work. Technical writing earns its precision through specifics.
⚠️
What could go wrong if they misread this?
This is the most important question. If the worst-case misreading of your document is "someone restarts the wrong pod during a P1 incident," that's a signal to be more specific — not more verbose.

The "Condescending" Problem — and How to Avoid It

Some engineers write vague documentation because they're worried about being too obvious. "My readers are senior engineers — they'll know what I mean."

This is a real concern, but it's pointed at the wrong target. Condescension in technical writing isn't caused by specificity — it's caused by explaining context the reader already has.

🚫
Condescending: explaining what the reader already knows
"Note: Kubernetes pods are the smallest deployable units in Kubernetes. Before restarting a pod, ensure you understand what a pod is and what restarting it will do to your application." — An SRE doesn't need this.
✅
Not condescending: being specific about what the reader doesn't know
"When restarting this pod, note that the model warm-up period is 45–60 seconds — unlike standard API pods. Requests during warm-up are queued, not dropped." — This is context the SRE needs, even if they've been doing this for years.

The difference: condescension explains shared knowledge. Specificity provides context that's genuinely new, environment-specific, or easy to assume incorrectly. Don't explain what Kubernetes is. Do explain that this particular service has a 60-second warm-up no one would expect.

A Checklist Before You Publish

Run through this before you mark any technical document as done. Click each item to check it off.

Clear Technical Writing — Pre-Publish Check
Every action step uses a specific command, value, or name — not a vague verb like "check," "fix," or "address"
Any system, service, endpoint, or environment mentioned by name — not "the system" or "the service"
If the reader needs to make a judgment call, I've told them what success and failure look like
If this is an alert, ticket, or incident report: who to escalate to and under what conditions
If AI generated any part of this: I've verified every specific claim, number, and command against the actual system
I've explained context that is specific to this environment — not general knowledge the reader already has
🧠 Quick Check
An on-call SRE receives an AI-generated alert summary that says: "Performance anomaly detected. Elevated response times observed in the production environment. Immediate investigation recommended." The SRE spends 18 minutes investigating before finding the actual problem in an unrelated service. What is the primary failure in this alert?
📌 Article Summary

What to Remember

Clear technical writing is not about simplifying complexity — it's about removing every reason someone could misread what you wrote. Technical readers fill in gaps with assumptions and keep moving. Those assumptions cause incidents.

Condescension comes from explaining what the reader already knows — not from being specific. Name the service, give the exact command, say what success looks like. That's not talking down to engineers; that's respecting their time.

AI-generated documentation is often vague in exactly the ways that matter most — specific values, system names, thresholds, and next steps. When you review AI-generated text, your job is to add the specificity the AI left out. Approving a vague AI draft transfers accountability for that vagueness to you.
