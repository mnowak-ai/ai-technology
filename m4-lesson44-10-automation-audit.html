Lesson 4.4 — Responsible Automation

The Automation Audit: How to Review an Existing Workflow and Find Its Weaknesses

You will inherit automated workflows you didn't build. This is guaranteed in any tech role that involves automation. This article gives you a structured process for auditing those workflows — knowing where to look, what questions to ask, and what warning signs to flag before they become incidents.

Why Inherited Automations Are Riskier Than Ones You Build

When you build an automation yourself, you understand its design assumptions, its known limitations, and the edge cases you couldn't handle. When you inherit one, you get the workflow without any of that context. The automation looks like a black box: it runs, it produces outputs, and you have no way to know whether it handles unusual inputs correctly, whether the data it's reading is still reliable, or whether the person who built it tested it properly.

An inherited automation that's been running for 18 months without incident might be genuinely well-built — or it might have a silent failure mode that just hasn't been triggered yet. An audit tells you which it is.

The most dangerous automation is the one that's been running for a long time with no problems — because nobody has looked at it since deployment, nobody knows what assumptions it makes, and nobody will notice when those assumptions stop being true.

The Six Audit Domains — Where to Look

A thorough automation audit covers six areas. Click each to see the specific questions to ask and what you're looking for.

1
🗺️
Design Understanding — Do you know what this automation actually does?
▾

Before you can evaluate an automation's risks, you need to understand its design: what triggers it, what actions it takes, what conditions it evaluates, and what data it reads and writes. If you can't answer these questions, that's already a finding — undocumented automations are a risk in themselves.

Check
Can you describe in plain English exactly what this automation does, step by step? If not, start by building that description — trace the flow from trigger to final action.
Check
What systems does it read from and write to? Which users, records, or processes are affected by its outputs?
🚩 Red flag
Nobody on the current team can explain what the automation does. Original builder has left. No documentation exists. This is an immediate priority — you're running an unknown process in production.
Check
Was the automation built for a business process that has since changed? Are the rules it implements still correct for current operations?
⚠️ Common finding: The automation was built for a process that was later modified. The automation rules were never updated and now reflect old business logic — but nobody noticed because outputs "look right" in aggregate.
2
📊
Performance Data — What do the run logs actually show?
▾

Every automation platform maintains a run history. This is the most objective record of what the automation has actually been doing — not what you think it's been doing. Pull 90 days of run history and examine it systematically.

Check
What is the success rate over the past 90 days? Has it been stable, improving, or declining?
🚩 Red flag
Success rate below 90%, or a visible downward trend even from a high starting point. Any automation above 5% failure rate deserves investigation.
Check
What do the failed runs have in common? Examine 10+ failed runs and look for patterns — same day of week, same input type, same step, same error message.
🚩 Red flag
Failed runs cluster around the same step or same error type, indicating a systematic issue rather than random failures.
Check
Has run duration changed over time? Is it taking significantly longer than it used to? (Often signals upstream API slowness or growing data volumes.)
⚠️ Common finding: Run success rate looks fine at 94% — but examination of failed runs reveals they all involve records with a specific characteristic (amounts over $10K, records from a specific department, tickets submitted via mobile) that the automation handles incorrectly.
3
✅
Output Quality — Are the outputs actually correct?
▾

A run marked "Success" means every step executed without throwing an error. It does not mean the output is correct. Verification requires checking actual outputs against ground truth — comparing what the automation produced to what should have been produced.

Check
Sample 20–30 records that the automation processed. Manually verify that each one was handled correctly — correct category, correct data, correct routing. What is the actual accuracy rate?
🚩 Red flag
Accuracy noticeably lower than expected, or accuracy varies significantly by input type or category.
Check
Are there patterns in the outputs? Does the automation systematically over-route or under-route certain categories? Does it produce blank fields in some records?
🚩 Red flag
Outputs look "mostly fine" in aggregate but have consistent quirks — always blank for a specific field, always routes a specific user's tickets to the wrong queue.
✅ What to look for in a healthy audit: Sampled outputs consistently match what manual processing would have produced. Blank fields or routing quirks either don't exist or have a known, acceptable explanation.
4
🛡️
Human Oversight — Is there meaningful review, or is it theater?
▾

Many automations have nominal human review steps — but whether those steps involve genuine evaluation or just rubber-stamping is a critical distinction. An 8-second average review time on a task that requires comparing AI output to source documents is theater, not oversight.

Check
Does a human review step exist? For what types of outputs or decisions?
🚩 Red flag
No human review step exists for outputs that are high-stakes, irreversible, or drive significant decisions.
Check
If a review step exists — what is the average time reviewers spend on each item? What is the correction rate (% of items where reviewers change the AI's output)?
🚩 Red flag
Very low average review time (under 15 seconds for complex tasks) or near-zero correction rate — both indicate rubber-stamping rather than genuine review.
Check
Is there an escalation path — a way for reviewers to flag an item for additional review or manager attention if something seems wrong?
⚠️ Common finding: A review step exists and the audit shows a 100% approval rate over 6 months. This is actually more concerning than a 98% approval rate — it suggests reviewers are approving everything without catching any errors, which means the review step isn't functioning as a quality check.
5
🔔
Monitoring & Alerting — Will you know when it breaks?
▾

The most critical question in an automation audit: if this automation starts failing silently, how long before someone notices? If the answer is "when a user reports a problem," the monitoring is inadequate.

Check
Are failure notification alerts configured? Who receives them? How quickly?
🚩 Red flag
No failure notifications. The only way failures are detected is manual log review or user complaints.
Check
Are there output sanity checks — automated checks that flag when outputs are outside expected ranges (e.g., "alert if weekly report shows >15% change in any metric week-over-week")?
🚩 Red flag
No output sanity checks. The automation can produce systematically wrong results with no alert firing.
Check
When were the authentication connections last verified? Are any approaching expiration?
✅ Minimum monitoring standard: Failure notifications configured → named person or team → response time expectation defined. Plus at minimum one output sanity check that would catch systematic wrong results.
6
⚠️
Risk Assessment — What's the worst case if this fails silently?
▾

The final domain is a consequence assessment: given what you now know about this automation's design, performance, oversight, and monitoring — what is the worst plausible failure scenario, and how quickly would it be detected?

Check
If the automation produced incorrect outputs for 2 weeks without detection, what would the impact be? (Quantify if possible: how many records affected, what downstream decisions made on bad data, what business or customer impact.)
Check
Given the current monitoring, how long would a silent failure realistically run before detection?
🚩 Red flag
High-impact consequences (financial decisions, security monitoring, customer-facing outputs) + long detection time (no monitoring = "until a user reports it") = highest-priority audit finding.
⚠️ The risk matrix: High impact + long detection time = urgent remediation. Low impact + fast detection = acceptable risk. Use this to prioritize which audit findings to address first.

Prioritizing What to Fix — The Risk Matrix

An audit will typically surface multiple findings. Use the risk matrix to prioritize: which findings deserve immediate action vs. planned remediation vs. monitoring?

🔴 Immediate
High impact + long detection time
Fix before the next business day. Examples: no monitoring on a financial automation, no HITL on security-critical routing, undocumented automation in production with no named owner.
🟠 Planned
High impact OR long detection time (not both)
Address within 2 weeks. Examples: review step exists but appears to be rubber-stamping, output sanity checks missing but failure notifications are configured.
🟢 Monitor
Low impact AND fast detection
Log and watch. Examples: documentation is thin but the automation owner knows how it works, training data is 12 months old but the category distribution hasn't changed significantly.
🧠 Quick Check
During an automation audit, you discover that a report generation workflow has been running for 9 months with no output sanity checks, no failure notifications, and a success rate of 99.8%. The reports are used by leadership for quarterly planning. A colleague argues: "The 99.8% success rate proves it's working fine — no audit findings needed here." What's wrong with this argument?
📌 Article Summary

What to Remember

An automation audit covers six domains: design understanding (do you know what it does?), performance data (what do the run logs show?), output quality (are outputs actually correct?), human oversight (is review genuine or rubber-stamping?), monitoring and alerting (will you know when it breaks?), and risk assessment (what's the worst case if it fails silently?). Each domain has specific questions and specific red flags.

Run success rate is not output accuracy. An automation with 99.8% run success rate can be producing systematically wrong outputs in every successful run — if the underlying data changed, an API format shifted, or a logic error produces wrong results without throwing an error. Output quality auditing requires sampling actual outputs and comparing them to ground truth, not checking whether runs completed.

Prioritize audit findings using the risk matrix: high-impact outputs + long detection time = immediate remediation. The most dangerous inherited automation is one that runs successfully in the background, produces decision-driving outputs, and has no monitoring — because when its assumptions become wrong, nobody will know until the damage is done.
