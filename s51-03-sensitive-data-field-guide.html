Lesson 5.1 — Security & Privacy Risks

What Is Sensitive Data? A Field Guide for Tech Workers Using AI

Before you can protect sensitive data, you have to be able to recognize it. This article builds the mental checklist that tech professionals use every day — so you can spot what's off-limits before it ends up in a prompt.

Why This Is Harder Than It Sounds

If someone asked you, "Is a patient's social security number sensitive?" you'd say yes immediately. That's obvious. But in your actual workday, the questions aren't that clear.

Is a list of usernames sensitive? What about an error log that includes IP addresses? What about code that connects to a database — the code itself, not the data inside? What about an internal Slack message your manager sent you about a client's complaint?

These are the real questions that tech workers face when they consider using AI tools. And the answer is almost always: yes, treat it as sensitive. This article explains why — and gives you a clear framework for making the call quickly.

The test isn't "Is this obviously sensitive?" The test is: "Would my company be comfortable if this appeared on an external server they don't control?" If the answer is no, or even maybe — that's your answer.

The Five Categories of Sensitive Data in Tech Roles

Here are the five types of sensitive data that tech workers encounter most often — and what makes each one protected:

👤
Personally Identifiable Information (PII)
Protected by law in almost every country and industry
🔴
Names, addresses, phone numbers, email addresses
Any information that can identify a specific person — even something as small as a name combined with a job title — counts as PII.
Example: A list of user accounts with names and emails
🔴
Social security numbers, government IDs, date of birth
These are high-risk identifiers. Even one field like a date of birth, combined with other data, can enable identity theft.
Example: A user database that includes birthdates
🔴
IP addresses and device identifiers
In many legal frameworks (especially in Europe under GDPR), an IP address is considered personal data because it can be traced to an individual or household.
Example: Application logs that include user IP addresses
💻
Proprietary Code and Intellectual Property
Protected by your employment contract and trade secret law
⚠️
Source code for products your company sells or uses internally
The code your company writes to build its products is a trade secret — legally protected proprietary information. Sharing it externally, even to debug it, can violate your NDA (non-disclosure agreement).
Example: The authentication module for your company's app
⚠️
Internal algorithms and business logic
How your product decides to do something — its pricing logic, recommendation engine, fraud detection rules — is often the most valuable intellectual property your company has.
Example: The formula that determines a user's risk score
⚠️
Database schemas and system architecture
Even without actual data in it, the blueprint of how your database is structured reveals how to attack it — and is usually considered proprietary information.
Example: Table names, column structures, foreign key relationships
🔑
Credentials and Access Keys
The most immediately dangerous category — never share these anywhere
🔴
Passwords and authentication tokens
If you paste code that contains a hardcoded password or session token, that credential is now outside your control. Change it immediately if this happens.
Example: password="MySecretPass123" in a config file
🔴
API keys
An API key is like a digital key to a system. If someone gets your company's AWS API key or Stripe secret key, they can make requests as if they were your company — often with real financial consequences.
Example: STRIPE_SECRET_KEY=sk_live_abc123 in an .env file
🔴
SSH keys and certificates
These grant access to servers and secure systems. A stolen SSH key can give an attacker direct access to your company's infrastructure.
Example: A private key file accidentally included in a code snippet
🏢
Internal System Details
Information that reveals how your company's systems are built and connected
🔵
Internal server names and URLs
Internal hostnames, internal IP addresses, and staging environment URLs tell attackers how your network is structured — and often have weaker security than public-facing systems.
Example: https://internal-staging.companyname.local
🔵
Error messages and stack traces with real paths
An error log might seem boring — but it often contains real file paths, server names, framework versions, and line numbers that reveal exactly how your systems are configured.
Example: Error at /home/app/production/payments/stripe_handler.py:47
👥
Customer and Client Data
Protected by contract, NDA, and often by law
🟣
Customer usage data, purchase history, account details
Any data your company collects from its customers is protected under your terms of service, your contracts, and usually multiple privacy laws. You have a legal obligation to keep it secure.
Example: A CSV of customer orders you're analyzing for a report
🟣
Client project details and communications
If your company works with external clients, their project details, strategies, and internal information are typically covered by an NDA. Sharing them — even to an AI — may be a breach of contract.
Example: Internal notes from a client strategy call

The "Is It Sensitive?" Quick Check

When you're not sure whether something is safe to put into an AI tool, run it through these questions. Answer honestly — the point is to protect yourself.

🔍 Sensitivity Check Tool
What are you thinking about pasting into an AI tool?
Does it contain any real names, email addresses, phone numbers, or IDs that belong to a real person?
Does it contain any code from your company's actual codebase — not example code you wrote from scratch?
Does it contain any passwords, API keys, internal URLs, or access credentials?
Does it come from a client, a customer, or an internal meeting — rather than being something you created yourself?
🧠 Quick Check
A developer wants to use an AI tool to help optimize a slow database query. The query references a table called "user_transactions" and includes a WHERE clause with a real customer's account ID. Which parts of this are sensitive?
📌 Article Summary

What to Remember

Sensitive data isn't just social security numbers and medical records. For tech workers, it also includes source code, database schemas, API keys, internal server details, and anything that comes from customers or clients.

The five main categories to watch for: PII (personal information), proprietary code and IP, credentials and access keys, internal system details, and customer/client data. Any of these in a prompt is a red flag.

When in doubt, use the "would my company be comfortable seeing this on an external server?" test. If you hesitate for even a second, the answer should be: don't paste it. Use dummy data instead.

Credentials — passwords, API keys, SSH keys — are in a category of their own. If you ever accidentally paste one of these into an AI tool, treat it as compromised and rotate it immediately. Don't wait to see if anything happens.
