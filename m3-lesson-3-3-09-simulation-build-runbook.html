📋 Corrected Rollback Procedure
ROLLBACK PROCEDURE ⚠ BEFORE STARTING: Confirm diagnostic steps in Section 2 are complete. Step 1. Roll back to v3.1 specifically (not just "previous") kubectl set image deployment/inference-api \ inference-api=fraud-detector:v3.1 -n ml-prod Why this command (not 'rollout undo'): 'rollout undo' rolls back to the immediately previous deployment, which may not be v3.1 if another change was deployed between v3.1 and v3.2. Always specify the exact target version. Step 2. Monitor rollback progress kubectl rollout status deployment/inference-api -n ml-prod Expected: "deployment 'inference-api' successfully rolled out" This takes 3–5 minutes. Do not proceed to Step 3 until this confirms. IF STEP 2 FAILS: Pods stuck in Pending or CrashLoopBackOff: → Do NOT retry. Escalate to ml-oncall immediately (PagerDuty: ml-p1-escalation). → While waiting: scale inference-api to 0 replicas to stop serving bad responses: kubectl scale deployment/inference-api --replicas=0 -n ml-prod Step 3. Confirm rollback version kubectl get deployment inference-api -n ml-prod \ -o jsonpath='{.spec.template.spec.containers[0].image}' Expected output: fraud-detector:v3.1 Step 4. Verify error rate recovery Wait 3 minutes, then check: curl https://inference.internal/metrics | grep error_rate Expected: error_rate < 0.5% (baseline) Check Datadog: ml-inference-perf dashboard → "Error Rate" panel → Target: green (< 1%). If still yellow/red after 5 min, escalate. Step 5. Confirm memory utilization is stable kubectl top pods -n ml-prod -l app=inference-api Expected: Memory < 1GB per pod (v3.1 baseline is ~600MB) Step 6. Update incident ticket Add rollback timestamp. Do NOT close ticket yet — leave open for 48-hour monitoring. Post in #ml-incidents: "v3.2 rolled back at [time UTC]. v3.1 live. Monitoring for 48h."

Section 5
Post-Rollback Actions & Review Cycle
Locked
📖 What you're writing
The runbook isn't over when the service recovers. Post-rollback actions capture what happens next: monitoring period, handoff documentation, and how/when to attempt redeployment. This section is almost always missing from AI-generated runbooks — and its absence is what causes the same incident to happen again.

Write the post-rollback and review cycle section. Include: monitoring period, what to watch for, when to attempt redeployment of v3.2, and runbook review trigger.

📋 Model Answer
POST-ROLLBACK MONITORING (48 hours) Watch for (check every 4 hours or set alert): • inference-api error rate (Datadog: ml-inference-perf) — alert if > 1% • Pod memory utilization — alert if > 1.2GB per pod (v3.2's problem started at 1.4GB) • Batch scoring queue depth — alert if > 300 items (normal ceiling: 200) If any of the above trigger during the 48-hour window: → This is likely a different issue not related to v3.2. Use the general inference-api incident runbook. → Do NOT re-rollback — v3.1 is already running. REDEPLOYMENT OF v3.2 REQUIREMENTS Do NOT attempt to redeploy v3.2 until ALL of the following are true: □ Root cause of the memory leak is identified and fixed (owner: ML team) □ Load test has been run at production batch size (64, not 128) — see P1-1115 action item □ ML team lead has approved the fix for deployment □ Deployment is scheduled during off-peak hours (not Thursday–Sunday) □ This runbook has been updated to reflect any changes in the rollback procedure Expected timeline for v3.2 re-deployment: 5–7 business days (ML team will confirm). RUNBOOK: inference-api — Model Version Rollback (v3.2 → v3.1)
Section 1 — Trigger Condition
Use when ALL of the following are true: • inference-api error rate > 5% for 3+ consecutive minutes • Current running model is fraud-detector:v3.2 • No other recent change explains the error rate If only some conditions are met → use general inference-api incident runbook (§2) first.
Section 2 — Diagnostic Steps
1. Confirm error rate > 5%: curl https://inference.internal/metrics | grep error_rate 2. Confirm v3.2 is running: kubectl get deployment inference-api -n ml-prod -o jsonpath='{.spec.template.spec.containers[0].image}' 3. Check pod memory: kubectl top pods -n ml-prod -l app=inference-api (expect > 1.4GB) 4. Check rollout history: kubectl rollout history deployment/inference-api -n ml-prod If all 4 confirm → proceed. If any inconclusive → escalate before proceeding.
Section 3 — Rollback Procedure
1. kubectl set image deployment/inference-api inference-api=fraud-detector:v3.1 -n ml-prod 2. kubectl rollout status deployment/inference-api -n ml-prod (wait for success) IF FAILS → scale to 0 replicas + page ml-p1-escalation immediately 3. Confirm: kubectl get deployment ... -o jsonpath='{...image}' → expect fraud-detector:v3.1 4. Wait 3 min, verify error rate < 0.5% 5. Verify memory < 1GB per pod 6. Update incident ticket. Leave OPEN for 48h. Post in #ml-incidents.
Section 4 — Escalation Path
Escalate immediately if rollback fails, error rate > 2% after 10 min, any CrashLoopBackOff, or any doubt about trigger condition. ML issues → PagerDuty: ml-p1-escalation Infra issues → PagerDuty: infra-p1-escalation > 30 min unresolved → notify @c.park in Slack
Section 5 — Post-Rollback & Review
Monitor 48h: error rate, pod memory (alert > 1.2GB), queue depth (alert > 300). v3.2 redeploy requires: root cause fixed + load test at batch-64 + ML lead approval + off-peak scheduling. Runbook update: after every incident involving inference-api. Review every 90 days.
