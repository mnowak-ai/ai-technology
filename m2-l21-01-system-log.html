Lesson 2.1 · Using AI to Diagnose Problems

What Is a System Log, Really? A Plain-Language Guide for Anyone New to IT Environments

System logs are the foundational evidence source for every technical diagnosis — and every AI diagnostic tool you will ever use. Before AI diagnosis makes sense, you need to understand what logs are, what they contain, and how professionals actually read them.

The Best Analogy for a System Log

Imagine a security camera that recorded every single thing that happened in a building — not video, but a written record. Every door that opened, every light that turned on, every person who badged in, every alarm that triggered, every system that malfunctioned. Timestamped, in order, with a note about what happened and how serious it was.

That is a system log. A log is a timestamped, chronological record of events inside a technical system. Every time something happens — a service starts, a user logs in, a database query runs, a connection fails, a threshold is exceeded — the system writes a line to the log. That line records what happened, when it happened, how serious it was, and which part of the system it came from.

When something goes wrong, the log is the first place a tech professional looks. It's the evidence trail that answers the question: what actually happened, in what order, right before things broke?

A system log is not a report about what went wrong. It's a complete record of everything that happened — which means the answer to why something went wrong is almost always in there, if you know how to read it.

Severity Levels: The Urgency Code in Every Log

Every log line has a severity level — a label that tells you how serious the recorded event is. Learning to read severity levels is one of the most important foundational skills in technical work, because severity is what separates background noise from signals that need attention.

DEBUG
Diagnostic Detail
Very detailed technical information. Usually only visible when debugging is turned on. Not shown in normal operations.
DB-Main: Query executed in 12ms
INFO
Normal Operations
Things are working as expected. Services starting, users logging in, tasks completing successfully.
Service 'UserAuth' started successfully
WARN
Watch This
Something unusual happened, but the system is still functioning. May indicate a developing problem.
High latency detected on API gateway
ERROR / CRITICAL
Something Failed
A specific operation failed (ERROR) or the entire service is unresponsive (CRITICAL). Requires immediate attention.
Connection timed out to DB-Main — FAILED

In practice, a busy production system generates thousands of log lines every hour. The vast majority are INFO — normal operations. WARN lines appear occasionally and are worth monitoring. ERROR and CRITICAL lines are what trigger investigation. Reading logs effectively means filtering to the right severity level for what you're trying to understand.

The Six Types of System Logs You'll Encounter

Not all logs are the same. Different types of logs record different things about different parts of a system. Click each log type to see what it captures and what it looks like:

📋
Application Logs
What your software is doing
Application logs capture events inside a specific piece of software — a web application, a database, an API service. They record things like user authentication attempts, query execution times, errors thrown by the code, and service start/stop events. These are the first logs you'd check if a specific application is misbehaving.
[10:45:15] WARN API-Gateway — High latency: 2400ms (threshold: 500ms) [10:45:18] ERROR DB-Main — Connection timed out [Retry 1/3] [10:45:27] CRIT DB-Main — Service unresponsive
🖥️
System / OS Logs
What the operating system is doing
System logs capture events at the operating system level — CPU and memory usage, disk read/write operations, hardware errors, kernel events, and system-level security events. These are useful when a problem might originate in the infrastructure layer rather than the application layer.
[10:44:55] INFO kernel — CPU usage: 43% [10:45:10] WARN kernel — CPU usage: 89% (spike detected) [10:45:28] CRIT kernel — CPU usage: 98% — throttling active
🔒
Security / Access Logs
Who accessed what and when
Security logs record authentication events, authorization decisions, login attempts (successful and failed), and access to sensitive resources. These are the primary evidence source for investigating security incidents — unauthorized access, credential stuffing attacks, or insider threats.
[10:40:01] INFO Auth — User jsmith logged in (IP: 192.168.1.10) [10:41:33] WARN Auth — Failed login attempt (IP: 203.0.113.42) [3 of 5] [10:41:44] ERROR Auth — Account locked after 5 failed attempts (IP: 203.0.113.42)
🌐
Network Logs
What's moving across the network
Network logs capture traffic flowing through routers, firewalls, and load balancers — which IP addresses are connecting, which ports are being used, how much data is moving, and whether connections are being accepted or rejected. Essential for diagnosing connectivity problems and security threats.
[10:45:14] INFO Firewall — Connection accepted: 10.0.0.5 → DB-Main:5432 [10:45:18] WARN Firewall — Connection timeout: 10.0.0.5 → DB-Main:5432 [10:45:22] ERROR Firewall — High packet loss detected: 23% (threshold: 1%)
🗄️
Database Logs
What's happening inside the database
Database logs are a specialized form of application log that record query execution, transaction commits and rollbacks, lock wait times, connection pool status, and storage operations. When performance problems relate to data access — slow queries, deadlocks, connection exhaustion — database logs are where you look.
[10:44:50] INFO PostgreSQL — Query executed: SELECT * FROM users (8ms) [10:45:05] WARN PostgreSQL — Slow query detected: JOIN orders (2340ms) [10:45:18] ERROR PostgreSQL — Max connections reached (100/100)
🔄
Event / Audit Logs
What changed and who changed it
Audit logs record deliberate actions taken by people or automated systems — configuration changes, deployments, permission modifications, data exports. These are critical for diagnosing problems that followed a change: "it broke right after the 10:30 deployment" is a hypothesis that audit logs can confirm or refute.
[10:30:02] INFO Deploy — Version 2.1.3 deployment started by j.smith [10:30:44] INFO Deploy — Database migration applied (schema v42 → v43) [10:31:00] INFO Deploy — Deployment completed successfully

The Most Important Skill: Reading Logs for Sequence

Individual log lines tell you what happened. The sequence of log lines tells you why it happened. A WARN that appears five minutes before a CRITICAL error is a completely different diagnostic situation from a WARN that appears after one. The order matters as much as the content.

This is why experienced tech professionals don't just Ctrl+F for "ERROR" in a log file and start there. They read the surrounding context — what was happening before the error, what events led up to it — because the cause of a problem usually appears in the log before the problem itself becomes visible.

Why AI Diagnostic Tools Start With Logs

When an AI diagnostic tool like Splunk, Datadog, or Google Cloud Operations analyzes a system problem, logs are its primary evidence source. AI can process thousands of log lines per second, identify patterns across multiple log sources simultaneously, and surface correlations that would take a human analyst hours to find manually.

But here is the critical thing to understand: AI reads the same log data a human reads. It just reads more of it, faster. This means everything you just learned about log structure — timestamps, severity levels, source identifiers, message content, and sequence — is exactly what AI is analyzing. When AI tells you "the database connection failure was likely caused by a schema migration," it reached that conclusion by reading the same log sequence you just practiced on.

⚡
What AI does with logs better than humans
Processes hundreds of thousands of log lines across multiple systems simultaneously. Identifies patterns — like "this error sequence always precedes this type of failure" — from historical data. Correlates events across application logs, system logs, network logs, and database logs at the same time. Surfaces the most anomalous events for human review.
AI's speed advantage is real and significant
🧠
What humans do with logs better than AI
Understand operational context that isn't in the logs — a scheduled maintenance window, a known infrastructure quirk, a recent configuration change that wasn't formally logged. Recognize when a log pattern that looks like a known problem is actually a different problem in disguise. Make judgment calls about which hypothesis is most consistent with what the team knows about the system.
Human judgment completes what AI surfaces

📌 Article Summary

What to Remember

A system log is a timestamped, chronological record of everything that happens inside a technical system. Every log line contains four key components: a timestamp, a severity level (DEBUG/INFO/WARN/ERROR/CRITICAL), a source identifier, and a message. All four matter for diagnosis.

Reading logs for sequence — not just for individual error lines — is what separates effective technical diagnosis from symptom chasing. The cause of a problem usually appears in the log before the visible failure, often as a WARN entry that preceded the ERROR. Skipping to the error means potentially missing what caused it.

AI diagnostic tools analyze the same log data a human reads — just faster and across more sources simultaneously. Understanding log structure is what lets you evaluate whether AI's interpretation of that data is correct. You cannot verify an AI root cause suggestion if you don't understand the log evidence it's based on.
