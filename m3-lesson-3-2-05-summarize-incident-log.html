Lesson 3.2 — Summarizing Technical Information

How to Summarize an Incident Log for Three Different Audiences

A 6-hour AI inference outage. A raw incident log. Three audiences who each need something completely different from it. Here's the full walkthrough — step by step.

Start With the Raw Material

Before you can summarize an incident log, you need to read it fully — not skim it. Below is a realistic incident log for a 6-hour AI inference service degradation. Read it once before moving to the summarization steps.

Step-by-Step: How to Summarize This Log

1
Extract the five required facts first

Before you write a single word of any summary, pull the five required elements from the raw log. Do this once — you'll use different parts for each audience's summary.

Your five extracted facts
What happened: Inference API degradation — error rate reached 34%, batch queue backed up to 4,200 items. Caused by oversized embedding layer in v3.2.

Who was affected: All users of features powered by the inference API. Recommendation feature specifically affected by rollback.

Current status: RESOLVED — v3.1 live as of 04:15 UTC. Recommendation feature temporarily unavailable.

Root cause: v3.2 embedding layer 3x larger than v3.1, causing GPU memory pressure at production batch sizes. Load test didn't catch it because test environment used different batch size config (128) than production (64).

What's next: (1) Fix embedding layer size in v3.2 before re-deploy. (2) Align test and production batch size configs. (3) Investigate why load test environment gap existed.
2
Identify what each audience needs from those facts

Now map your extracted facts to each audience's job. What does each reader need to do after reading their summary — and which facts enable that action?

Audience mapping
SRE postmortem team: needs full root cause, exact timeline, and specific action items to fix the problem and prevent recurrence. Needs the batch config gap detail — that's the key finding.

Engineering leadership / manager: needs to know the service is restored, what decision was made (rollback), which feature was dropped and for how long, and what the team is doing to fix v3.2 before re-deploy.

Product team: needs to know the recommendation feature is down, why, when it will be back, and what they should tell users or adjust in roadmap planning.
3
Write each summary — in order of urgency

Write the most time-sensitive summary first — usually the operational or stakeholder one, since those audiences may need to act or communicate immediately. The postmortem can wait until the team reconvenes.

📌 Article Summary

What to Remember

The incident log summarization process has three steps: extract the five required facts first (what, who, status, cause, next steps), map those facts to each audience's job, then write each summary in order of urgency. Doing the extraction step once saves you from re-reading the log three times.

The same fact can be appropriate in one summary and noise in another. The test config gap belongs in engineering summaries because engineers need to fix it. It doesn't belong in the product team summary because the product team can't act on it and it doesn't change any product decision. Deciding what's noise for each audience is the core skill.

Write the most time-sensitive summary first. In incident situations, the operational and stakeholder summaries are often needed within minutes of resolution — they can't wait for the postmortem. The detailed technical postmortem is important but not urgent in the same way.
