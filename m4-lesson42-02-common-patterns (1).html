Lesson 4.2 — Pattern Recognition & Insights

Common Patterns Every Tech Worker Should Know How to Spot: Errors, Spikes, Dips, and Bottlenecks

The video listed four patterns on one slide. Each one is a learnable shape in data — with a recognizable signature, a set of likely causes, and a different appropriate response. This article gives each pattern its full treatment.

Why Patterns Are the Language of Tech Data

Experienced tech workers develop a kind of data fluency — they look at a chart or log stream and immediately recognize shapes they've seen before. Not because they memorized rules, but because they've encountered these patterns enough times to have intuitions about what they mean and what usually comes next.

Each of the four patterns below has a recognizable shape, a set of common causes, and appropriate responses. Learning to distinguish them — rather than seeing all data events as generically "something happened" — is the core skill of pattern recognition in tech.

The Four Patterns — Explore Each One

⚠️
Repeated Errors
The same error appearing multiple times in a short window — especially when it hasn't appeared before or is occurring at a higher rate than usual.
Error rate over time — notice the sudden onset and sustained elevation

A repeated error pattern has a distinct shape: a stable baseline period, followed by sudden onset of the same error appearing repeatedly. Key diagnostic questions: when did it start? (correlate with deployments or config changes), is it one error or multiple? (same message = systemic cause), is the rate stable or growing? (growing = situation still developing). In a ticket queue, this pattern looks like the same problem arriving in a burst from multiple users — treat the systemic cause first, not the individual tickets.

IT Operations
"NullPointerException in UserController" appears 0 times for 3 months, then 47 times in 2 hours after a deployment
A deployment-introduced bug. The sudden onset correlation with a specific deployment points directly to a code regression.
🤖 AI detects the onset correlation with the deployment timestamp and escalates the error rate automatically.
IT Support
3 "can't access email" tickets arrive within 10 minutes — the daily average for this category is 1
Three identical complaints in 10 minutes is a repeated error pattern in ticket form. Check the email server before routing individual tickets.
🤖 AI ticket anomaly detection flags when a category receives 3x its normal rate in a short window — before the queue fills with duplicates.
✅ Response
Check for a deployment, config change, or dependency failure correlating with the onset time
Determine whether the rate is stable or accelerating — accelerating requires faster escalation
If systemic, address the root cause first — don't process 34 individual VPN tickets before checking if the VPN itself is down
📈
Spikes
A sharp, sudden increase in a metric — traffic, CPU, ticket volume, error rate — that rises quickly, peaks, and either returns to baseline or stays elevated.
Two spikes with very different shapes and meanings

Spikes are the most common pattern in tech data — and the most misinterpreted. Key distinctions: an expected spike (known cause, returns to baseline — scheduled jobs, marketing campaigns), a sustained spike (doesn't return — capacity or configuration problem), and an unexplained spike (requires investigation). The metric that's spiking determines the severity: a traffic spike with flat error rate is often fine. The same traffic spike with a simultaneous error spike means the system is saturating.

IT Operations
CPU spikes to 95% for 4 minutes, then returns to 42%
Check whether it correlates with a scheduled job. If it always produces this spike at this time, it's noise. If it started causing request timeouts, it's a capacity signal.
🤖 AI distinguishes between spikes correlating with known scheduled events vs. unexpected spikes through time-series pattern matching.
IT Support
Help desk ticket volume spikes 400% in one afternoon
Always ask: what happened right before the spike? A product update, system outage, or company announcement. Spikes without obvious causes need investigation before routing decisions are made.
🤖 AI can compare spike onset time against deployment logs and calendar events to help correlate cause and effect faster.
✅ Response
First: is this spike expected? Check against scheduled jobs, known traffic patterns, recent events
Second: is it returning to baseline? A spike that doesn't return is a different problem than one that does
Check correlated metrics — a traffic spike without error rate change is usually fine; same spike WITH error rate change means saturation
📉
Dips
A sudden drop in a metric that should be continuously active — traffic, successful request rate, user activity — often indicating a system has stopped working or stopped being reachable.
Login success rate — a dip to zero is the most alarming pattern in tech monitoring

Dips are often more alarming than spikes — especially when a metric that should always be non-zero drops toward zero. A successful request rate dropping to zero means something stopped working entirely. The critical distinction: expected vs. unexpected. A dip at 3am during scheduled maintenance is expected. The same shape at 10:30am Tuesday is an outage. Context determines everything.

IT Operations
Server health check success rate drops from 100% to 0% suddenly
The server is unreachable, overloaded, or crashed. P1 incident — the service is down. Response time is measured in minutes, not hours.
🤖 AI monitoring detects the transition to zero within seconds and fires an immediate high-priority alert — far faster than any human scanning a dashboard.
Analytics
App daily active users drops 90% week-over-week
This magnitude almost never reflects real user behavior. It usually means a tracking failure — the analytics code stopped firing. Investigate the measurement system before drawing business conclusions.
🤖 AI flags statistically impossible drops — a 90% single-week DAU decline is outside any normal variation and warrants immediate data quality investigation.
✅ Response
First, rule out measurement failure — the thing being measured may have broken, or the measurement itself may have broken
For business-hours dips in operational metrics, escalate immediately — a 10am traffic dip to zero is almost certainly an outage
Document start time precisely — outage duration has SLA and customer communication implications
🚧
Bottlenecks
A point in a workflow where one step is slower than everything before and after it — causing a queue to build upstream and stalling downstream work.
IT support workflow — identifying the bottleneck step by average processing time per stage

A bottleneck is different from the other three patterns — it's not an error state. It's a structural problem: one step that processes slower than everything surrounding it, causing a queue to form in front of it and everything downstream to wait. In the diagram above, Integration Support takes 18.7 hours while every other step takes under 1 hour. Adding capacity to any other step in this workflow won't help — work will still pile up in front of Integration Support. The only fix is addressing the constraint itself.

Bottlenecks appear in IT support queues (one team always backlogged), deployment pipelines (one test step taking 45 minutes while others take 2), data pipelines (one transformation can't keep pace), and approval workflows (one decision-maker is a dependency for many processes).

IT Support
Integration tickets average 18.7 hrs; everything else averages under 4 hrs
The fix isn't routing faster or adding general capacity — it's specifically addressing Integration Support capacity or skill. The bottleneck must be named and addressed at its root.
🤖 AI-assisted workflow analytics can automatically identify the bottleneck step by comparing average processing time per stage — something very hard to see in raw ticket data.
DevOps / Platform
CI/CD pipeline: tests take 42 min; every other step takes under 3 min
Developers wait 45+ minutes per deployment because one step dominates. This directly affects release velocity and the ability to ship fixes quickly. Parallelizing tests directly addresses the constraint.
🤖 Pipeline analytics can show per-step timing trends over time — revealing when a 15-minute test suite has grown to 42 minutes as coverage expanded.
✅ Response
Identify the specific constraint step — don't optimize the whole workflow; optimize the bottleneck
Measure the queue in front of it: how many items waiting, how long is the wait? This quantifies the impact
Diagnose the cause: is the step slow because it's understaffed, technically inefficient, or handling work it wasn't designed for?

📌 Article Summary

What to Remember

Repeated errors have a sudden onset after a stable baseline and often correlate with a specific cause. In ticket queues, they appear as multiple tickets with the same symptom arriving in a burst — treat the systemic cause first, not the individual tickets.

Spikes require context: expected vs. unexpected, returning vs. sustained. Always check the spiking metric AND correlated metrics — a traffic spike without an error rate change is usually fine; the same traffic spike with an error rate spike means the system is saturating.

Dips in metrics that should be continuously non-zero are the most alarming pattern — always rule out measurement failure first. Bottlenecks are structural: one step slowing everything else. They require targeted intervention at the constraint, not optimization of the surrounding workflow.
