Lesson 2.1 · Using AI to Diagnose Problems

A Day in the Life of an IT Operations Analyst Using AI Diagnostic Tools

Abstract concepts become clear when you see them in action. This article follows one analyst through a real shift — the alerts, the AI outputs, the judgment calls, and the moments where the analyst's thinking matters more than the tool's suggestion.

Meet Jordan — IT Operations Analyst, 3 Years In

Jordan works on a mid-size IT operations team for a financial services company. Their environment runs a mix of cloud and on-premise infrastructure: payment processing systems, customer portal applications, internal databases, and a network of servers handling thousands of transactions per hour. The team uses Splunk as their primary AI-assisted monitoring and diagnostic platform.

Jordan's shift is 7AM–3PM. During that window, the AI platform will generate dozens of alerts, flag several anomalies, and probably surface two or three diagnostic suggestions that will need to be evaluated. Jordan's job is not to click "apply fix" on everything the AI surfaces. Jordan's job is to think through every AI suggestion and decide whether it's correct, incomplete, or wrong before anything happens to a production system.

AI makes the analyst faster. The analyst makes the AI useful. Neither is effective without the other in a high-stakes production environment.

07:00 — Shift start
Morning handoff & dashboard review
🤝 Human + AI
Jordan logs in and pulls up the overnight summary. The AI platform has processed 4.2 million log entries since the previous shift and flagged 7 anomalies for review. Three were acknowledged by the night team and are in monitoring status. Four need fresh eyes.
🤖 AI Overnight Summary
Anomalies flagged: 7 ✅ Acknowledged (3): Network latency spike [resolved], Storage warning [monitoring], Auth timeout [monitoring] ⚠️ Needs review (4): → Payment API response time +340% above baseline (03:17–03:45) → DB-Prod-02 disk I/O unusual pattern (ongoing) → Unusual login volume from service account SVC-BATCH (04:22) → Memory usage trending up: App-Server-07 (6hr trend)
🧠 Jordan's First Decision
Jordan doesn't immediately act on any of these. The payment API spike ended at 03:45 — nearly 4 hours ago — and the system is currently normal. Jordan checks whether 03:17–03:45 is during scheduled batch processing (it is). That context isn't in the AI summary. Jordan marks that one "likely expected — verify with batch schedule" and moves to the active concerns: the disk I/O pattern and the memory trend.
07:45 — First active investigation
DB-Prod-02 disk I/O anomaly
🤖 AI-led initial analysis
Jordan dives into the disk I/O anomaly on DB-Prod-02. The AI has already analyzed the pattern and generated a diagnostic suggestion.
🤖 AI Diagnostic Output
DB-Prod-02 Disk I/O Analysis Anomaly: Read operations 280% above 30-day baseline (ongoing 9hrs) Pattern match: Similar to index rebuild events (Historical match: 78%) Possible Cause A: Automated index rebuild running unscheduled (Confidence: 72%) Possible Cause B: Table scan query without index (Confidence: 45%) Possible Cause C: Backup process consuming I/O (Confidence: 31%) Recommended: Check scheduled task logs for index rebuild activity
🧠 Jordan's Evaluation
72% confidence on the index rebuild — plausible. Jordan checks the scheduled task log. No index rebuild is scheduled for this server. Jordan then checks recent database schema changes: there was a new index added to the customer transaction table yesterday afternoon. That index may still be building on the production replica. Jordan confirms this in the DBA change log — not in the AI's data sources. The AI was directionally right (it's index-related) but missed the actual cause (a new index still completing, not an unscheduled rebuild). Jordan notes this, updates the ticket with the correct finding, and confirms no action needed — the index build will complete on its own.
09:20 — New critical alert fires
Payment gateway: 5xx error rate spike
🤖 AI immediate triage
An alert fires mid-morning: the payment gateway is returning 5xx errors at 12% of requests — well above the 0.5% normal rate. This is customer-facing and business-critical. Jordan has roughly 5 minutes before the incident needs to be escalated to senior leadership.
🤖 AI Real-Time Triage
🔴 CRITICAL — Payment Gateway 5xx Spike Error rate: 12.3% (baseline: 0.4%) Onset: 09:18:42 (2 min ago) Correlated events: → App-Server-03 memory: 94% (critical threshold) → App-Server-03 GC pause time: 4,200ms (normal: <100ms) → No deployment in last 6 hours Primary hypothesis: Memory exhaustion on App-Server-03 causing request failures Confidence: 81% Suggested action: Restart App-Server-03 application service
🧠 Jordan's Critical Decision
81% confidence and a clear correlated cause. Jordan is tempted to restart immediately. But first: is App-Server-03 behind a load balancer? Yes — there are 4 app servers. Jordan pulls App-Servers 01, 02, 04: all normal. Restarting 03 will temporarily remove it from the pool — acceptable. But Jordan also checks: did any scheduled job start at 09:18? Yes — a large report generation job started at 09:17. Jordan flags this: the memory spike may be caused by the report job, not by a systemic memory issue. Restart would fix the symptom; canceling or throttling the report job fixes the root cause. Jordan cancels the report job — memory begins releasing within 90 seconds. Error rate drops to 0.8% and continues falling. No restart needed. Incident resolved in 6 minutes.
11:00 — Ongoing trend monitoring
App-Server-07 memory trend — proactive action
👤 Human-led judgment
The memory trend on App-Server-07 that was flagged overnight hasn't triggered an alert yet — but Jordan checks in on it.
🤖 AI Trend Projection
App-Server-07 Memory Usage Trend Current: 71% (28.4GB / 40GB) 6-hour trend: +2.1% per hour Projected threshold crossing (85%): ~6.7 hours Historical pattern match: Gradual leak pattern (68% match) Possible cause: Memory leak in running process No alert threshold crossed yet — monitoring
🧠 Jordan's Proactive Judgment
The AI is right that this looks like a leak pattern — but "6.7 hours to threshold" puts it right in the middle of peak evening usage (5–9PM). Jordan decides not to wait for the alert. They identify the process with highest memory growth: a scheduled analytics job that started at 01:00 and is still running 10 hours later (it normally completes in 4). Jordan checks with the data team — the job is stuck in a loop on a dataset that's larger than expected. The data team kills and restarts it with a corrected dataset. Memory begins releasing. The potential evening alert is avoided entirely. This is the kind of proactive catch that AI can flag but only human judgment can act on — the AI doesn't know what "peak evening usage" means for this organization.
14:30 — End of shift documentation
Incident summaries & handoff notes
🤝 Human + AI
Jordan uses the AI platform to generate draft incident summaries for each investigation. The AI pulls timestamps, log evidence, and AI diagnostic outputs into a structured report. Jordan reviews and edits each one.
🤖 AI Draft — DB-Prod-02 Incident Summary
Incident: DB-Prod-02 Disk I/O Anomaly Duration: Overnight (9+ hours) AI Diagnosis: Possible unscheduled index rebuild (72% confidence) Action Taken: Monitored; no intervention Resolution: Auto-resolved
🧠 Jordan's Edit — What the AI Draft Missed
Jordan edits the summary to reflect the actual finding: "AI diagnosis was directionally correct (index activity) but missed the actual cause: a new index added yesterday to the customer transaction table is still completing its build on the production replica. AI's data sources did not include the DBA change log. Recommend: ensure DBA change log is integrated into AI platform data sources to improve future diagnosis accuracy on this pattern." This kind of feedback loop — human analyst correcting AI output and flagging the data gap — is how organizations improve their AI diagnostic tools over time. It's a critical part of Jordan's job that doesn't show up in any tool's feature list.

Five Questions Jordan Asks About Every AI Diagnostic Output

🧠 Jordan's Mental Checklist
Applied to every AI suggestion before any action is taken
1
What data did the AI analyze? — Does it have access to the change logs, scheduled task calendars, and team-specific context that might explain this differently? What is it missing?
2
Is the confidence score based on a pattern match to a genuinely similar situation? — Or is the AI matching superficial similarities between this situation and a past failure that had a different underlying cause?
3
Does the suggested action address a symptom or a root cause? — A restart might fix the symptom in 2 minutes. Finding what caused the condition might take 20 minutes. Which is the right call given current severity and business impact?
4
What happens if this action makes things worse? — Can it be reversed? Does it affect other systems? Who needs to be informed before I act?
5
Is the AI blind spot here worth documenting? — If AI missed something important because it didn't have access to a data source, that's feedback that can improve future diagnostic accuracy. Log it.

📌 Article Summary

What to Remember

AI diagnostic tools handle the high-volume, pattern-matching work that no human can do manually at speed — processing millions of log entries, correlating events across systems, surfacing anomalies. The analyst's job is to apply the contextual knowledge, organizational history, and critical judgment that AI tools don't have access to.

High AI confidence doesn't mean the suggested action is correct — it means the pattern strongly matches historical data. The analyst must always ask: does the AI have the full picture? Scheduled jobs, change logs, team context, and organizational knowledge frequently change the right course of action even when AI is directionally correct about the technical symptom.

Documenting AI blind spots is part of the analyst's professional responsibility — not just a nice-to-have. When AI misses a cause because it lacked access to a data source (like the DBA change log), that finding, documented and fed back, improves the AI tool's future effectiveness. The analyst who does this consistently makes their team's AI tools more accurate over time.
