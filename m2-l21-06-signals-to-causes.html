Lesson 2.1 · Using AI to Diagnose Problems

From Signals to Possible Causes: How AI Diagnoses a System Problem Step by Step

The diagnostic flow from raw signals to proposed root causes involves specific steps that both AI and human experts work through together. This article walks through the complete process with a real database timeout scenario — showing exactly what data goes in, what AI surfaces, what confidence scores mean, and what the human expert does next.

Diagnosis Is a Process, Not a Button

When people first see AI diagnostic tools in action, a common misconception is that diagnosis happens in one step: data goes in, answer comes out. The reality is more structured — and understanding the structure is what allows you to participate intelligently in the diagnostic process rather than simply accepting or rejecting AI output wholesale.

The complete diagnostic flow has five distinct stages. AI handles some of them; humans handle others; some are genuinely collaborative. Walk through the complete process below using a real scenario: a database connection timeout incident in a production e-commerce environment.

AI accelerates the signal-to-hypothesis step by an order of magnitude. The hypothesis-to-decision step still requires a human. Knowing which stage you're at determines what you should do next.

Stage 1 — AI
Raw Signal Ingestion: What the AI Is Reading
The AI diagnostic system continuously ingests all available signals from the production environment. At the time of the incident, here is the complete signal picture it is analyzing:
LOGS (Last 15 minutes): 10:43:12 WARN Network throughput dropped: 80Mbps → 12Mbps 10:44:58 INFO Application server started normally 10:45:15 WARN API gateway latency: 2,847ms (threshold: 500ms) 10:45:18 ERROR DB-Main connection timeout [1/3] 10:45:21 ERROR DB-Main connection timeout [2/3] 10:45:24 ERROR DB-Main connection timeout [3/3] FAILED 10:45:27 CRIT Database service unresponsive ALERTS (Active): 🔴 DB Server CPU: 98% (threshold: 90%) — NEW 🔴 API Gateway: 5xx error rate spike — NEW 🟡 Storage /data: 85% capacity — ACKNOWLEDGED METRICS (Current vs. Baseline): DB Server CPU: 98% (baseline: ~40%) API Gateway Mem: 28/32GB (baseline: ~16GB) Net Throughput: 8 Mbps (baseline: ~80 Mbps)
What's happening at this stage: AI is not interpreting yet. It's collecting the complete picture simultaneously — logs, alerts, and metrics across all connected systems. A human analyst might search logs for the database errors. AI ingests everything at once, including the network throughput drop 2 minutes before the first log entry.
Stage 2 — AI
Pattern Recognition: What the AI Matched
After ingesting all signals, AI compares the current data pattern against its training data — millions of historical incidents from similar environments. It identifies three significant pattern matches:
PATTERN MATCH #1 (High Confidence): Sequence: API gateway latency spike → DB connection timeout cluster → This sequence matches 847 historical incidents in training data → 73% resolved to "Database Deadlock / Resource Exhaustion" → 18% resolved to "Network-induced connection timeout" PATTERN MATCH #2 (Medium Confidence): Metric combination: DB CPU 98% + Memory 28/32GB simultaneously → Matches "resource saturation" pattern → Consistent with heavy query load or memory leak scenarios PATTERN MATCH #3 (Low Confidence): Network throughput drop preceding application errors → Less common pattern — intermittent packet loss can cause → connection timeouts, but usually produces different error sequence
Key insight: The AI flagged the network throughput drop at 10:43 — 2 minutes before the first log entry. A human analyst searching logs for "database" or "connection timeout" would likely have missed this entirely. The temporal relationship between the network drop and the subsequent errors is a diagnostic clue the AI surfaced automatically.
Stage 3 — AI
Cause Hypotheses: What AI Proposes
Based on the pattern matches, AI generates a ranked list of possible root causes with confidence scores and supporting evidence for each:
AI-Generated Possible Causes
Database Deadlock — Concurrent high-priority transactions locking each other; consistent with CPU spike and connection timeout cluster
85%
Network Congestion / Packet Loss — Throughput drop at 10:43 preceded errors; intermittent packet loss can produce connection timeouts
60%
Application Memory Leak — API gateway memory at 28/32GB is elevated; gradual leak could be causing resource pressure
30%
⚠️ These are probabilistic starting points for investigation, not confirmed diagnoses. The 85% score means this pattern most frequently led to this cause in historical data — not that it is definitely the cause in this specific incident.
Stage 4 — HUMAN EXPERT
Human Evaluation: Applying Context AI Doesn't Have
The human expert now reviews the AI's proposed causes against operational context — information the AI doesn't have access to. This is where the diagnostic process shifts from AI to human judgment:
Human expert review notes: [Database Deadlock — 85%] ✓ Plausible: CPU spike is consistent with deadlock scenario ✓ Plausible: Connection timeout cluster matches deadlock pattern → Check: Were there any large batch jobs or migrations running today? → Found: Yes — a data migration job started at 10:40am (5 min before incident) → This significantly increases likelihood of deadlock as root cause [Network Congestion — 60%] ? Note: We did have a planned network maintenance window ending at 10:45 → The throughput drop at 10:43 may be residual from maintenance, not a cause → Need to check network team's maintenance logs to confirm → If maintenance explains the network anomaly, this cause becomes less likely [Memory Leak — 30%] ? API gateway memory has been elevated for 3 days (known pre-existing issue) → This predates the incident — less likely to be the acute cause → Still needs addressing, but separately from this incident
What the human added that AI couldn't: The data migration job that started at 10:40am. The planned network maintenance window. The 3-day history of elevated gateway memory. All of this context significantly changes the interpretation of AI's proposed causes. Without it, the 60% network cause might look as likely as the 85% deadlock cause. With it, the deadlock cause becomes the clear priority to investigate first.
Stage 5 — HUMAN + AI
Actionable Resolution: Confirmed Cause, Targeted Fix
With context applied, the human expert confirms the root cause and implements a targeted fix — not a generic restart, but a specific remediation addressing the confirmed cause:
CONFIRMED ROOT CAUSE: Database Deadlock Evidence: Migration job at 10:40am created concurrent transactions locking critical tables simultaneously with production queries. IMMEDIATE ACTIONS TAKEN: 1. Killed the deadlocked transactions (restored service in 4 minutes) 2. Terminated and rescheduled the migration job for off-peak hours (2am) 3. Confirmed database returned to normal CPU levels (~38%) PREVENTIVE ACTIONS: 1. Updated migration scheduling policy: no large jobs during business hours 2. Added pre-check: migration jobs must acquire table locks before starting 3. Set up separate alert for concurrent long-running transactions DOCUMENTED ROOT CAUSE: Migration job competing with production queries for table locks → database deadlock → connection timeout cascade → API gateway failure
Result: Service restored in 4 minutes (vs. 35+ minute average for similar incidents without AI-assisted diagnosis). Root cause addressed specifically — no generic restart, no recurrence of the same category of incident. The network maintenance noise and pre-existing memory issue identified as separate items for follow-up.

What Human Evaluation Added That AI Couldn't

👤 The Four Types of Context Humans Provide
Why the evaluation step isn't optional
1
Recent operational changes: The data migration job started 5 minutes before the incident. AI doesn't know your change calendar. Humans do — and connecting a recent change to a current failure is often the fastest path to root cause confirmation.
2
Known pre-existing conditions: The gateway memory had been elevated for 3 days — a pre-existing issue that predated the incident. Without that context, AI's 30% memory leak suggestion could mislead investigation. Humans track the history; AI sees the current snapshot.
3
Scheduled events and maintenance windows: The network maintenance window ending at 10:45 likely explained the throughput anomaly — making the 60% network cause significantly less likely. That scheduled event was invisible to AI but immediately relevant to human evaluation.
4
System-specific architecture knowledge: Knowing which tables the migration job was accessing, which production queries compete for the same locks, and what "deadlock" typically looks like in this specific environment — all of this domain knowledge belongs to the human expert, not the AI.

📌 Article Summary

What to Remember

The complete signal-to-resolution flow has five stages: signal ingestion (AI), pattern recognition (AI), cause hypothesis generation (AI), human evaluation (human), and resolution (human + AI). The first three happen in seconds; the fourth requires operational context that only the human expert has. Skipping Stage 4 means acting on statistical patterns without the specific context that makes those patterns meaningful in your actual environment.

Operational context — recent changes, scheduled events, pre-existing conditions, architecture-specific knowledge — is what transforms AI's probabilistic hypotheses into confirmed diagnoses. In the walkthrough scenario, a 5-minute-old data migration job completely changed the interpretation of AI's suggestions. No AI system has access to your change calendar. Human evaluation provides it.

A well-executed AI-assisted diagnosis reduces MTTR (Mean Time to Resolution) dramatically — in the scenario, from 35+ minutes to 4 minutes — not because AI replaced human judgment, but because it handled the signal analysis and pattern recognition stages that would otherwise consume most of the time available for diagnosis. Human judgment then operated on complete, organized information rather than raw data.
